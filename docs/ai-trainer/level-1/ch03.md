# 第三章：团队管理与项目落地

> 分值占比：20%-30% | 重要程度：★★★★★

## 考情分析

本章是高级技师级别的核心管理能力考核内容，涉及项目管理方法论、团队建设与协作、成果交付与验收等管理技能。考试以案例分析题和方案设计题为主，要求考生具备全局视角和实操经验。

---

## 3.1 项目管理

### 3.1.1 AI 项目生命周期

AI 项目与传统软件项目相比，具有更强的探索性和不确定性。完整生命周期包括：

```
需求定义 → 数据准备 → 模型开发 → 系统集成 → 部署上线 → 运营迭代
    ↑                                                           |
    └──────────────── 反馈闭环 ←────────────────────────────────┘
```

**各阶段关键活动：**

| 阶段 | 关键活动 | 里程碑 |
|------|----------|--------|
| 需求定义 | 业务调研、问题定义、可行性分析 | PRD 文档 |
| 数据准备 | 数据采集、清洗、标注、质检 | 标注数据集 |
| 模型开发 | 算法选型、训练调优、离线评估 | 模型报告 |
| 系统集成 | API 封装、前后端对接、测试 | 集成测试报告 |
| 部署上线 | 环境搭建、灰度发布、监控配置 | 上线发布 |
| 运营迭代 | 效果监控、Bad Case 分析、模型迭代 | 迭代报告 |

### 3.1.2 项目管理方法论

**瀑布模型 vs 敏捷方法：**

| 维度 | 瀑布模型 | 敏捷方法（Scrum） |
|------|----------|-------------------|
| 适用场景 | 需求明确、变更少的项目 | 需求不确定、快速迭代的项目 |
| 特点 | 阶段分明，文档驱动 | 迭代开发，持续交付 |
| 风险管理 | 后期才发现问题 | 早期持续验证 |
| AI 项目适配 | 部分适用（模型部署阶段） | **推荐**（数据+模型迭代） |

**AI 项目更适合敏捷/混合方法的原因：**
- 数据质量问题和边界 case 通常在后期才被发现
- 模型效果需要不断调优，不可能一次到位
- 业务需求在项目推进中可能发生变化
- 需要快速验证假设，降低试错成本

**敏捷项目管理关键实践：**

| 实践 | 说明 |
|------|------|
| Sprint 规划 | 每 2-4 周一个迭代周期 |
| 每日站会 | 15 分钟同步进度和障碍 |
| 迭代评审 | 演示可工作的增量 |
| 回顾会议 | 总结改进点 |
| 待办列表 | 优先级排序，持续更新 |

### 3.1.3 风险管理

AI 项目特有的风险及应对：

| 风险类型 | 风险描述 | 应对策略 |
|----------|----------|----------|
| 数据风险 | 数据质量差、标注不一致、数据泄露 | 数据质量审计、标准化标注流程、脱敏 |
| 技术风险 | 模型效果不达标、性能瓶颈 | 多方案并行、性能基准测试 |
| 进度风险 | 数据准备耗时远超预期 | 数据准备预留充足时间 |
| 合规风险 | 算法备案、数据合规 | 早期引入法务/合规团队 |
| 落地风险 | 用户不接受、效果无法持续 | 早期用户参与、持续运营 |

::: tip 高频考点
1. AI 项目生命周期的六个阶段
2. 瀑布模型 vs 敏捷方法在 AI 项目中的适用性
3. AI 项目特有风险：数据风险、技术风险、合规风险
4. 风险管理的基本流程：识别→评估→应对→监控
5. Sprint 敏捷开发的核心实践
:::

::: warning 重要提醒
AI 项目成功率低的主要原因：低估数据准备的工作量和难度。数据准备通常占 AI 项目 60%-80% 的时间。
:::

## 3.2 团队协作

### 3.2.1 AI 团队角色

典型的 AI 产品团队包含以下角色：

| 角色 | 职责 | 所需技能 |
|------|------|----------|
| 产品经理 | 需求定义、产品设计、用户研究 | 业务理解、需求分析 |
| AI 算法工程师 | 模型设计、训练调优 | 数学基础、编程、算法 |
| 数据标注员 | 数据标注、质量控制 | 细心、理解标注规范 |
| 数据工程师 | 数据处理管道、特征工程 | SQL、ETL、数据建模 |
| 后端工程师 | 服务开发、系统集成 | 后端开发、API 设计 |
| 前端工程师 | 界面开发、交互设计 | 前端开发、UI/UX |
| 测试工程师 | 功能测试、性能测试、模型测试 | 测试方法论、自动化测试 |
| 运维工程师 | 部署运维、监控告警 | Docker、K8s、监控工具 |
| 项目经理 | 进度管理、资源协调 | 项目管理、沟通协调 |

### 3.2.2 跨职能协作

AI 项目涉及多个职能团队，高效的跨职能协作是关键：

**协作痛点：**
- 算法工程师与产品经理之间：需求理解不一致
- 算法工程师与数据标注员之间：标注质量争议
- 算法工程师与后端工程师之间：模型部署集成问题

**协作优化方案：**

| 方案 | 实施方式 | 效果 |
|------|----------|------|
| 统一语言 | 建立业务术语表和指标词典 | 减少沟通误解 |
| 标准接口 | 模型服务使用统一 API 格式（如 OpenAPI） | 降低集成成本 |
| 联合评审 | 算法、产品、数据定期联合评审 | 及时发现偏差 |
| 工具链统一 | 统一的 ML 实验跟踪（MLflow）、数据标注平台 | 提高协作效率 |
| 文档规范 | 模型卡（Model Card）、数据表（Datasheet） | 信息共享透明化 |

### 3.2.3 沟通管理

**向上沟通（对管理层）：**
- 使用业务语言，避免过多技术细节
- 聚焦 ROI、里程碑和风险
- 提供决策选项和推荐方案

**平级沟通（对合作团队）：**
- 明确接口和交付物
- 及时同步变更
- 主动解决问题

**向下沟通（对团队成员）：**
- 明确目标和期望
- 提供必要资源和支持
- 及时反馈和认可

::: tip 高频考点
1. AI 团队的核心角色和职责
2. 数据准备在 AI 项目中占比最大（60%-80%）
3. 跨职能协作的关键：统一语言、标准接口、联合评审
4. 模型卡（Model Card）和数据集表（Datasheet）的概念
5. 不同层级的沟通策略
:::

## 3.3 成果交付

### 3.3.1 模型交付标准

AI 模型的交付不仅是代码，还需要完整的文档和验证：

**模型交付物清单：**

| 交付物 | 内容 | 目的 |
|--------|------|------|
| 模型文件 | 训练好的模型权重 | 部署使用 |
| API 接口 | REST/gRPC 接口 | 系统集成 |
| 模型报告 | 性能指标、混淆矩阵、Bad Case 分析 | 效果验证 |
| 模型卡 | 模型用途、性能、局限性、伦理考量 | 文档归档 |
| 训练脚本 | 复现训练的代码 | 可复现性 |
| 依赖说明 | 环境配置、依赖版本 | 部署环境搭建 |

### 3.3.2 验收标准

AI 模型的验收标准应包括：

| 维度 | 验收内容 | 标准示例 |
|------|----------|----------|
| 功能正确性 | 基本功能是否正常工作 | 核心场景准确率 ≥ 95% |
| 性能指标 | 延迟、吞吐量 | 响应时间 < 200ms |
| 稳定性 | 异常输入处理 | 错误率 < 0.1% |
| 鲁棒性 | 边界 case 处理 | 极端情况不崩溃 |
| 合规性 | 是否符合法规要求 | 数据脱敏、算法备案 |
| 文档完整性 | 文档是否齐全 | 模型卡、API 文档齐全 |
| 可维护性 | 代码是否可维护 | 代码审查通过、有单元测试 |

### 3.3.3 知识转移

模型交付后，还需要进行知识转移：

```
知识转移内容：
├── 技术培训 — 模型原理、调参方法、常见问题
├── 运维手册 — 监控指标、故障排查流程
├── 数据流程 — 数据更新、再训练流程
└── 使用文档 — API 调用、示例代码
```

**知识转移的步骤：**
1. 制定培训计划：确定培训对象和内容
2. 技术培训：讲解模型原理和调优方法
3. 文档移交：提供完整的技术文档和运维手册
4. 交接会议：正式移交项目，确认各方理解
5. 支持期：设置过渡支持期（通常 1-3 个月）

### 3.3.4 成果复盘

每个 AI 项目完成后应进行复盘：

| 复盘维度 | 核心问题 |
|----------|----------|
| 目标达成 | 是否达到预期业务目标？差距原因？ |
| 过程回顾 | 哪些做法有效？哪些需要改进？ |
| 经验沉淀 | 可复用的经验有哪些？ |
| 团队成长 | 团队能力提升了什么？ |

::: tip 高频考点
1. 模型交付物清单（模型文件、API、报告、模型卡、脚本、依赖说明）
2. 验收标准的七个维度
3. 知识转移的五个步骤
4. 项目复盘的四个维度
5. 模型卡（Model Card）的概念和作用
:::

::: warning 验收关键
模型效果达到标准只是验收的一部分，文档完整性、可维护性和知识转移同样是高级技师考核重点。
:::

## 考点速记

::: tip 高频考点
1. AI 项目生命周期：需求→数据→模型→集成→部署→运营
2. 数据准备占 AI 项目 60%-80% 的时间
3. 敏捷 vs 瀑布：AI 项目推荐敏捷/混合方法
4. AI 特有风险：数据风险、技术风险、合规风险
5. AI 团队角色：产品、算法、数据、工程、测试、运维
6. 跨职能协作：统一语言、标准接口、联合评审
7. 模型交付物：模型文件、API、报告、模型卡、脚本、依赖说明
8. 验收标准：功能、性能、稳定性、鲁棒性、合规、文档、可维护性
9. 知识转移：培训→文档→移交→支持
10. 项目复盘：目标、过程、经验、成长
:::

::: warning 核心公式/原则
- **数据准备时间 ≈ AI 项目总时间的 60%-80%**
- **AI 成功 = 高质量数据 × 合适算法 × 工程能力 × 业务理解**
- **灰度发布：1-5% → 10-20% → 50% → 100%**
:::
