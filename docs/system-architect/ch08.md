# 第八章：云计算与容器

> 分值占比：5%-10% | 重要程度：★★★

## 考情分析

云计算与容器技术是系统架构设计师考试的新增重点，主要考查云计算基本概念、虚拟化技术、容器技术、Kubernetes编排、云原生架构设计等。考试重点在于IaaS/PaaS/SaaS的区别、Docker容器原理、Kubernetes核心组件与资源对象、云原生架构特征。综合知识题常涉及概念辨析，案例分析题可能涉及云原生系统架构设计。

---

## 8.1 云计算基础

### 8.1.1 云计算定义与特征

- **定义**：通过网络按需提供可伸缩的共享计算资源
- **NIST定义五大特征**：
  1. 按需自助服务
  2. 广泛的网络接入
  3. 资源池化
  4. 快速弹性伸缩
  5. 可度量的服务

### 8.1.2 云计算服务模型

| 服务模型 | 提供者 | 用户管理 | 示例 |
|---------|--------|---------|------|
| IaaS（基础设施即服务） | 虚拟化硬件 | 操作系统及以上 | AWS EC2、阿里云ECS |
| PaaS（平台即服务） | 运行环境 | 应用程序和数据 | AWS Elastic Beanstalk、Heroku |
| SaaS（软件即服务） | 完整应用 | 仅配置和使用 | Office 365、Salesforce |

**责任共担模型：**
- 云厂商：基础设施、物理安全
- 用户：操作系统、应用、数据安全

### 8.1.3 云计算部署模型

- **公有云**：第三方提供商，多租户，按需付费
- **私有云**：企业自建，独占资源，安全性高
- **混合云**：公有云+私有云混合，数据和应用可移植
- **社区云**：多个组织共享，支持特定社区

## 8.2 虚拟化技术

### 8.2.1 虚拟化概念

- **定义**：将物理资源抽象为多个虚拟资源的技术
- **类型**：
  - 服务器虚拟化：VMware、KVM、Xen
  - 存储虚拟化：SAN、NAS
  - 网络虚拟化：VLAN、SDN
  - 桌面虚拟化：VDI

### 8.2.2 服务器虚拟化

- **全虚拟化**：完全模拟硬件，无需修改OS（VMware、KVM）
- **半虚拟化**：需要修改OS内核以优化（Xen）
- **硬件辅助虚拟化**：CPU硬件支持（Intel VT-x、AMD-V）

### 8.2.3 容器虚拟化

- **对比虚拟机：**

| 对比项 | 虚拟机 | 容器 |
|--------|--------|------|
| 隔离级别 | OS级隔离 | 进程级隔离 |
| 资源开销 | 大（完整OS） | 小（共享内核） |
| 启动速度 | 慢（分钟级） | 快（秒级） |
| 镜像大小 | GB级 | MB级 |
| 性能 | 接近物理机 | 接近物理机 |

## 8.3 容器与Docker

### 8.3.1 Docker概述

- **定义**：开源容器引擎，简化应用部署
- **核心思想**：Build Once, Run Anywhere
- **三大核心概念**：
  - 镜像（Image）：只读模板，包含应用及其依赖
  - 容器（Container）：镜像的运行实例
  - 仓库（Repository）：存储和分发镜像

### 8.3.2 Docker镜像

- **分层存储**：镜像由多层组成，层间共享
- **Dockerfile**：定义镜像构建步骤
  - FROM：基础镜像
  - RUN：执行命令
  - COPY/ADD：复制文件
  - EXPOSE：声明端口
  - CMD/ENTRYPOINT：启动命令

### 8.3.3 Docker网络

- **Bridge模式**：默认，容器间通过虚拟网桥通信
- **Host模式**：容器共享宿主机网络栈
- **None模式**：容器无网络
- **Overlay模式**：跨主机容器网络

### 8.3.4 Docker存储

- **Volume**：Docker管理的持久化存储
- **Bind Mount**：挂载宿主机目录
- **tmpfs**：内存挂载，非持久化

### 8.3.5 Docker Compose

- **定义**：定义和运行多容器应用的工具
- **核心文件**：docker-compose.yml
- **功能**：服务定义、网络配置、卷挂载、启动顺序

## 8.4 Kubernetes（K8s）

### 8.4.1 Kubernetes概述

- **定义**：容器编排平台，自动化容器部署、扩缩容和管理
- **核心能力**：
  - 服务发现与负载均衡
  - 存储编排
  - 自动部署和回滚
  - 自动扩缩容
  - 自我修复

### 8.4.2 Kubernetes架构

- **控制平面（Control Plane）**：
  - **API Server**：所有组件交互的入口
  - **etcd**：分布式键值存储，保存集群状态
  - **Scheduler**：Pod调度到合适节点
  - **Controller Manager**：运行各种控制器

- **工作节点（Worker Node）**：
  - **Kubelet**：管理节点上的Pod生命周期
  - **Kube-proxy**：网络代理，实现Service抽象
  - **Container Runtime**：容器运行时（containerd、CRI-O）

### 8.4.3 Kubernetes核心资源

**Pod**：
- 最小部署单元，包含一个或多个容器
- 共享网络命名空间和存储卷
- 短暂存在，不保证持久性

**Deployment**：
- 管理Pod的副本集
- 支持滚动更新和回滚
- 声明式配置

**Service**：
- 为Pod提供稳定访问端点
- 类型：ClusterIP、NodePort、LoadBalancer
- 实现负载均衡

**ConfigMap & Secret**：
- ConfigMap：存储配置数据
- Secret：存储敏感信息（密码、密钥）

**Volume & PVC/PV**：
- Volume：Pod级别的存储
- PV：集群级别的存储资源
- PVC：存储请求声明
- StorageClass：动态存储供给

### 8.4.4 Kubernetes调度

- **默认调度器**：基于节点资源、亲和性、污点等
- **调度流程**：过滤（Filter）→ 打分（Score）→ 绑定（Bind）
- **高级调度**：
  - 节点亲和性（Node Affinity）
  - Pod亲和性/反亲和性（Pod Affinity/Anti-Affinity）
  - 污点与容忍（Taints & Tolerations）
  - 节点选择器（Node Selector）

### 8.4.5 Kubernetes运维

- **Helm**：Kubernetes包管理器
- **Kustomize**：原生配置管理工具
- **监控**：Prometheus + Grafana
- **日志**：ELK Stack、Loki
- **CI/CD**：Argo CD、Jenkins X

## 8.5 云原生架构

### 8.5.1 云原生定义

- **CNCF定义**：云原生技术使组织能够在公有云、私有云和混合云等现代动态环境中，构建和运行可弹性扩展的应用。
- **核心技术**：容器、服务网格、微服务、不可变基础设施、声明式API

### 8.5.2 云原生架构特征

- **容器化**：应用打包为容器，环境一致性
- **微服务**：服务拆分，独立部署
- **服务网格（Service Mesh）**：
  - 代表：Istio、Linkerd
  - 功能：流量管理、可观测性、安全性
  - 架构：数据平面（Sidecar代理）+ 控制平面

- **不可变基础设施**：部署后不修改，更新通过替换实现
- **声明式API**：声明期望状态，系统自动调整
- **DevOps**：开发运维一体化，CI/CD流水线

### 8.5.3 云原生设计原则

- **无状态服务**：服务不保存会话状态，便于扩缩容
- **弹性设计**：自动扩缩容、熔断、限流、降级
- **可观测性**：日志、指标、链路追踪三位一体
- **自动化**：自动化部署、监控、故障恢复

### 8.5.4 Serverless架构

- **定义**：无需管理服务器，按实际使用付费
- **层级**：
  - IaaS：管理基础设施
  - PaaS：管理运行时
  - FaaS（函数即服务）：管理函数执行
  - BaaS（后端即服务）：托管后端服务

- **代表产品**：AWS Lambda、阿里云函数计算、腾讯云SCF

## 考点速记

::: tip 高频考点
1. 云计算三层：IaaS（基础设施）、PaaS（平台）、SaaS（软件）
2. 容器 vs 虚拟机：容器共享内核、进程级隔离、启动快、镜像小
3. Docker三大核心：镜像（只读）、容器（运行实例）、仓库（存储分发）
4. K8s核心资源：Pod（最小单元）、Deployment（副本管理）、Service（访问入口）
5. K8s控制平面：API Server（入口）、etcd（存储）、Scheduler（调度）、Controller Manager
6. 云原生核心：容器、微服务、服务网格、不可变基础设施
7. 服务网格：Istio/Linkerd，Sidecar代理模式
8. 声明式API：声明期望状态，系统自动调整
:::

## 重点难点

::: warning 重点难点
1. **虚拟化与容器**：虚拟化模拟硬件，容器共享OS内核；容器更轻量、启动更快，但隔离性弱于虚拟机
2. **K8s调度流程**：过滤节点（资源、亲和性等）→ 打分排序 → 绑定Pod到节点
3. **Service类型**：ClusterIP（集群内访问）、NodePort（节点端口）、LoadBalancer（负载均衡器）
4. **ConfigMap vs Secret**：ConfigMap存配置，Secret存敏感信息（Base64编码，非加密）
5. **云原生 vs 传统架构**：云原生强调自动化、弹性、不可变基础设施、微服务拆分
6. **Serverless优势**：按需付费、自动扩缩容、无需运维，但存在冷启动问题
:::
