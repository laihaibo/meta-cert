# 第六章：系统架构设计理论

> 分值占比：10%-15% | 重要程度：★★★★★

## 考情分析

系统架构设计是系统架构设计师考试的核心章节，主要考查软件架构风格、架构设计视图、架构评估方法、特定领域软件架构以及基于服务的架构设计。考试重点在于各种架构风格的特点与适用场景、4+1视图模型、ATAM/SAAM评估方法，以及SOA与微服务架构的设计原则。案例分析题常涉及架构模式选型、架构评估和微服务拆分。

---

## 6.1 软件架构风格

### 6.1.1 数据流风格

- **批处理风格（Batch Sequential）**
  - 数据按批次处理，各处理步骤顺序执行
  - 特点：简单、易于理解和实现
  - 适用：数据批处理、ETL场景

- **管道/过滤器风格（Pipes and Filters）**
  - 每个处理步骤封装为过滤器，通过管道连接
  - 特点：松散耦合、支持并发、可重用
  - 适用：编译器、信号处理、数据处理流水线
  - 缺点：不适合交互式应用

### 6.1.2 调用/返回风格

- **主程序/子程序风格（Main Program/Subroutine）**
  - 层次化调用结构，自顶向下分解
  - 特点：结构清晰，易于理解和维护
  - 适用：结构化程序设计

- **面向对象风格（Object-Oriented）**
  - 以对象为基本单元，封装数据和行为
  - 特点：封装、继承、多态
  - 适用：复杂业务系统

- **层次化风格（Layered）**
  - 系统分为多个层次，每层只能调用相邻下层
  - 典型：OSI网络模型、TCP/IP协议栈、Web应用分层
  - 特点：支持抽象、易于维护
  - 缺点：可能影响性能

### 6.1.3 独立构件风格

- **进程通信风格（Communicating Processes）**
  - 独立进程通过消息传递通信
  - 特点：并发执行、松耦合
  - 适用：分布式系统

- **事件驱动风格（Event-Driven）**
  - 构件通过事件进行通信，发布/订阅模式
  - 特点：高度解耦、异步处理、可扩展
  - 适用：GUI应用、消息中间件、监控系统

### 6.1.4 虚拟机风格

- **解释器风格（Interpreter）**
  - 解释器执行指令序列
  - 适用：脚本语言解释器、规则引擎

- **基于规则的系统（Rule-Based）**
  - 规则库 + 推理引擎
  - 适用：专家系统、决策支持系统

### 6.1.5 仓库风格

- **数据库风格（Database）**
  - 数据为中心，应用程序访问共享数据库
  - 特点：数据共享、数据一致性
  - 适用：信息管理系统

- **黑板风格（Blackboard）**
  - 知识源通过黑板共享数据
  - 适用：语音识别、图像处理、复杂问题求解

## 6.2 架构设计视图与4+1视图模型

### 6.2.1 4+1视图模型概述

Philippe Kruchten提出的4+1视图模型是软件架构设计的经典框架，从不同角度描述系统架构：

- **逻辑视图（Logical View）**：关注系统功能需求，描述系统的对象模型
- **开发视图（Development View）**：关注软件模块的组织和管理
- **进程视图（Process View）**：关注系统的并发和同步
- **物理视图（Physical View）**：关注硬件拓扑和部署
- **场景（Scenarios）**：用例场景，用于驱动和验证其他视图

### 6.2.2 各视图详细说明

**逻辑视图：**
- 主要元素：类、对象、关系
- 表示方法：类图、对象图
- 设计问题：抽象、封装、继承、多态
- 适用场景：面向对象设计、功能建模

**开发视图（模块视图）：**
- 主要元素：模块、包、层
- 表示方法：包图、组件图
- 设计问题：模块分解、接口定义、依赖管理
- 适用场景：项目组织、配置管理、构建系统

**进程视图：**
- 主要元素：进程、线程、通信
- 表示方法：活动图、时序图
- 设计问题：并发、同步、通信、死锁
- 适用场景：实时系统、分布式系统

**物理视图（部署视图）：**
- 主要元素：节点、连接、部署
- 表示方法：部署图
- 设计问题：硬件拓扑、网络通信、负载均衡
- 适用场景：系统部署、性能优化

### 6.2.3 视图映射与架构描述

- **逻辑视图到开发视图**：类到模块的映射
- **逻辑视图到进程视图**：对象到进程/线程的映射
- **开发视图到物理视图**：模块到节点的部署映射
- **场景**：串联各视图，验证架构设计

## 6.3 架构评估方法

### 6.3.1 SAAM（软件架构分析方法）

**目的：** 评估架构对特定质量属性的支持能力

**步骤：**
1. 确定评估的需求场景
2. 描述架构
3. 对场景进行分类（直接、间接、不相关）
4. 评估场景的互操作性
5. 形成评估结果

**特点：**
- 主要用于可修改性、可移植性评估
- 非破坏性评估，不改变架构
- 简单易用，适合早期评估

### 6.3.2 ATAM（架构权衡分析方法）

**目的：** 在多个质量属性之间进行权衡分析

**步骤：**
1. 确定评估参与者
2. 评估架构的商业目标
3. 确定架构方法
4. 确定场景
5. 评估架构方法
6. 识别风险点和敏感点
7. 进行权衡分析

**关键概念：**
- **敏感点（Sensitivity Point）**：影响特定质量属性的架构决策
- **权衡点（Tradeoff Point）**：同时影响多个质量属性的架构决策
- **风险点（Risk Point）**：可能导致架构问题的决策
- **非风险点（Non-risk Point）**：不会导致问题的决策

**输出：**
- 风险点清单
- 敏感点和权衡点清单
- 架构评估报告

### 6.3.3 其他评估方法

- **CBAM（成本效益分析方法）**：基于ATAM，增加经济角度分析
- **SAAMER（SAAM演进版）**：评估架构的可演化性

## 6.4 特定领域软件架构（DSSA）

### 6.4.1 DSSA基本概念

- **定义**：特定领域软件架构（Domain Specific Software Architecture）
- **特点**：
  - 针对特定领域（如航空、电信、金融）
  - 包含领域模型、参考需求、参考架构
  - 支持该领域内多个应用的开发

### 6.4.2 DSSA核心活动

1. **领域识别**：确定领域边界
2. **领域建模**：建立领域模型
3. **架构设计**：设计参考架构
4. **架构应用**：基于DSSA开发具体应用

### 6.4.3 DSSA与框架的区别

| 对比项 | DSSA | 框架 |
|--------|------|------|
| 关注点 | 领域特定的架构 | 通用的应用框架 |
| 粒度 | 宏观架构设计 | 具体实现框架 |
| 重用性 | 领域内多个应用 | 跨领域应用 |
| 示例 | 电信运营支撑系统 | Spring、.NET |

## 6.5 基于服务的架构

### 6.5.1 SOA（面向服务的架构）

- **核心概念**：
  - 服务：自包含、无状态的功能单元
  - 服务提供者：提供服务
  - 服务消费者：使用服务
  - 服务注册中心：服务目录

- **服务特征**：
  - 服务自治性：独立控制
  - 服务抽象性：隐藏实现细节
  - 服务可重用性：可被多个消费者使用
  - 服务无状态性：不保存会话状态

- **服务粒度**：粗粒度服务 vs 细粒度服务

### 6.5.2 微服务架构

- **核心特点**：
  - 单一职责：每个服务只做一件事
  - 独立部署：服务可独立发布和升级
  - 技术异构：不同服务可使用不同技术栈
  - 去中心化治理：各团队自主选择技术
  - 容错性：服务间隔离，避免级联故障

- **微服务 vs 单体架构**：

| 对比项 | 单体架构 | 微服务架构 |
|--------|---------|-----------|
| 部署 | 整体部署 | 独立部署 |
| 技术栈 | 统一 | 异构 |
| 团队协作 | 集中 | 分散 |
| 扩展性 | 整体扩展 | 按需扩展 |
| 故障隔离 | 差 | 好 |
| 开发复杂度 | 低 | 高 |
| 运维复杂度 | 低 | 高 |

- **微服务拆分原则**：
  - 按业务能力拆分
  - 按领域驱动设计（DDD）的限界上下文拆分
  - 单一职责原则
  - 闭包原则

- **微服务技术栈**：
  - 服务注册与发现：Eureka、Nacos、Consul
  - API网关：Spring Cloud Gateway、Kong
  - 配置中心：Spring Cloud Config、Apollo
  - 熔断限流：Hystrix、Sentinel、Resilience4j
  - 链路追踪：Zipkin、Jaeger、SkyWalking
  - 消息队列：RabbitMQ、Kafka

## 考点速记

::: tip 高频考点
1. 五大架构风格：数据流、调用/返回、独立构件、虚拟机、仓库
2. 管道/过滤器适合批处理流水线；事件驱动适合GUI和异步处理
3. 4+1视图：逻辑、开发、进程、物理、场景
4. ATAM重点：敏感点、权衡点、风险点、非风险点
5. SAAM vs ATAM：SAAM注重可修改性，ATAM注重多质量属性权衡
6. DSSA：特定领域、参考架构、领域模型
7. 微服务特点：单一职责、独立部署、技术异构、容错
8. SOA vs 微服务：SOA更偏企业级集成，微服务更偏敏捷和去中心化
:::

## 重点难点

::: warning 重点难点
1. **架构风格选择**：数据流适合批处理；调用/返回适合结构化；事件驱动适合GUI；仓库适合数据共享
2. **4+1视图应用**：逻辑视图关注功能，进程视图关注并发，开发视图关注模块组织，物理视图关注部署，场景贯穿验证
3. **ATAM评估流程**：商业动机→场景→架构方法→敏感点/权衡点→风险点→报告
4. **敏感点 vs 权衡点**：敏感点影响单一质量属性；权衡点影响多个质量属性（如安全性与性能的权衡）
5. **微服务拆分**：按领域驱动设计的限界上下文拆分，避免服务间过度调用
6. **分布式一致性**：CAP理论（一致性、可用性、分区容错性三选二）；BASE理论（基本可用、软状态、最终一致性）
:::
