# 案例分析

## 考试形式说明

- **题量**：共 4 道大题
- **时间**：90 分钟
- **满分**：75 分
- **题型**：每道大题包含 3~5 个小题，总分约 75 分

### 试卷结构

| 题号 | 题型 | 分值 | 考查重点 |
|------|------|------|----------|
| 第 1 题 | 系统架构设计 | 约 20 分 | 架构图绘制、模块划分、接口设计 |
| 第 2 题 | 质量属性场景分析 | 约 15 分 | 性能、可用性、安全性、可修改性 |
| 第 3 题 | 架构模式识别与应用 | 约 20 分 | 设计模式、架构风格、框架选型 |
| 第 4 题 | 新技术场景分析 | 约 20 分 | 微服务、云原生、大数据、AI 架构 |

### 评分要点

- **答案完整性**：覆盖题目要求的每个方面，不遗漏关键点
- **专业性**：使用规范的架构术语和设计原则
- **逻辑清晰**：条理分明，层次清楚
- **结合场景**：能够根据题目给出的业务场景进行针对性分析

::: tip 考点速记
- 案例分析是**综合题**，不是选择题，要求"写"出完整答案
- 每题平均用时约 22 分钟，建议先读问题再带着问题读案例描述
- 画图类题目（如架构图、数据流图）要标注关键组件和数据流向
- 遇到不会的题目，也要尽量写，写出相关术语和思路也能得分
:::

## 典型题型解析

### 题型一：系统架构设计

#### 题目特点

- 给出一个业务场景（如电商、金融、医疗等）
- 要求设计系统的整体架构
- 通常需要绘制架构图并说明设计理由

#### 解题框架

**1. 需求分析**
- 识别功能需求和非功能需求
- 明确用户角色和业务流程
- 估算系统规模（用户量、数据量、并发量）

**2. 架构分层设计**
```
应用层 → 接入层 → 服务层 → 数据层 → 基础设施层
```

**3. 关键设计点**
- **服务划分**：按领域边界拆分微服务
- **数据架构**：数据库选型、缓存策略、数据同步
- **技术选型**：中间件、框架、通信协议
- **部署架构**：容器化、负载均衡、高可用

**4. 非功能保障**
- 可用性：多可用区部署、故障转移
- 性能：缓存、CDN、异步处理
- 安全性：认证授权、数据加密、审计日志
- 可扩展性：水平扩展、服务化、消息队列

#### 答题技巧

- 先画出整体架构图（用 ASCII 字符或描述性文字表示层次）
- 对每个层次的关键组件进行说明
- 针对题目中提到的特殊需求（如高并发、大数据量）重点阐述解决方案
- 说明设计中的权衡（trade-off）

### 题型二：质量属性场景分析

#### 题目特点

- 给出一个系统场景
- 要求分析系统的某个质量属性（如性能、安全性、可用性）
- 可能要求提出改进方案

#### 质量属性六大类

| 质量属性 | 关注点 | 常见考查方式 |
|----------|--------|--------------|
| 性能 | 响应时间、吞吐量 | 如何优化系统性能 |
| 可用性 | 正常运行时间、容错 | 如何提高系统可用性 |
| 安全性 | 认证、授权、加密 | 如何保障系统安全 |
| 可修改性 | 模块独立性、扩展性 | 如何降低维护成本 |
| 可测试性 | 测试覆盖率、测试效率 | 如何设计可测试的架构 |
| 可用性（易用性） | 用户友好度 | 如何提升用户体验 |

#### 解题框架

**1. 场景描述**
- 明确刺激（stimulus）：什么条件触发质量属性变化
- 环境（environment）：系统在什么状态下运行
- 工件（artifact）：哪个部分受到影响
- 响应（response）：系统应如何响应
- 响应度量（response measure）：如何量化评估

**2. 问题分析**
- 当前架构存在什么问题
- 该问题导致什么后果
- 问题的根本原因是什么

**3. 改进方案**
- 提出具体的架构改进措施
- 说明每种措施的效果和代价
- 比较不同方案的优劣

#### 示例：性能优化分析

**场景**：某电商网站在促销活动期间，商品详情页的响应时间超过 5 秒，严重影响用户体验。

**分析框架**：
- **刺激**：促销期间高并发访问
- **环境**：生产环境，高峰期 QPS 达到 10000
- **工件**：商品详情页服务
- **响应**：响应时间超过 5 秒
- **响应度量**：P99 响应时间 > 5s

**改进方案**：
1. 引入 Redis 缓存热点数据（商品信息、库存状态）
2. 使用 CDN 加速静态资源（图片、CSS、JS）
3. 采用消息队列异步处理非核心逻辑（推荐、统计）
4. 数据库分库分表，读写分离
5. 前端优化：懒加载、骨架屏、减少重渲染

### 题型三：架构模式识别与应用

#### 题目特点

- 给出一个系统设计片段或代码片段
- 要求识别所使用的架构模式/设计模式
- 分析其优缺点及适用场景

#### 常见架构模式

**分层架构（Layered Architecture）**
- 表现层（UI Layer）
- 业务逻辑层（Business Logic Layer）
- 数据访问层（Data Access Layer）
- 数据库层（Database Layer）

**优点**：结构清晰，易于理解和维护
**缺点**：性能较差，层间依赖可能导致变更困难
**适用场景**：企业级应用、传统 Web 应用

**微服务架构（Microservices Architecture）**
- 服务独立部署
- 服务间通过 API 通信
- 每个服务有自己的数据库

**优点**：技术异构、独立扩展、团队自治
**缺点**：分布式系统复杂性、服务治理成本高
**适用场景**：大型复杂系统，需要快速迭代的业务

**事件驱动架构（Event-Driven Architecture）**
- 事件生产者发布事件
- 事件消费者订阅并处理事件
- 通过消息中间件解耦

**优点**：高度解耦、异步处理、可扩展
**缺点**：事件流难以追踪、调试困难
**适用场景**：实时处理、流式计算、IoT 系统

**CQRS（命令查询职责分离）**
- 命令（Write）和查询（Read）分离
- 使用不同的数据模型和存储
- 通过事件同步读写数据

**优点**：读写性能优化、模型针对性设计
**缺点**：复杂性增加、数据一致性维护成本高
**适用场景**：高并发读写场景、复杂业务规则

**六边形架构（Hexagonal Architecture）**
- 端口（Ports）定义接口
- 适配器（Adapters）实现接口
- 核心业务逻辑与外部依赖隔离

**优点**：测试友好、外部依赖可替换
**缺点**：学习曲线较陡
**适用场景**：需要高测试覆盖率的领域驱动设计

#### 解题技巧

- 准确识别模式名称和类别（架构模式 vs 设计模式）
- 说明该模式的核心思想
- 分析在当前场景下使用该模式的原因
- 讨论该模式的优缺点
- 提出可能的改进或替代方案

### 题型四：新技术场景分析

#### 题目特点

- 涉及新兴技术（如微服务、云原生、Serverless、AI 等）
- 要求分析技术选型、架构设计或技术趋势
- 结合实际业务场景进行分析

#### 云原生架构

**核心技术组件**：
- 容器化（Docker）：应用打包和运行环境标准化
- 容器编排（Kubernetes）：自动化部署、扩缩容、服务发现
- 服务网格（Istio/Linkerd）：流量管理、可观测性、安全
- 不可变基础设施：基础设施即代码（IaC）
- 声明式 API：通过声明期望状态来管理系统

**设计原则**：
- 构建可替换的组件而非固定架构
- 在流程中内置弹性（resilience）而非事后添加
- 自动化所有可自动化的操作
- 可观测性优先（日志、指标、链路追踪）

#### Serverless 架构

**适用场景**：
- 事件驱动的短期任务（如图片处理、数据转换）
- API 后端（API Gateway + Lambda/Function）
- 定时任务（如数据备份、报表生成）

**优势**：
- 无需管理服务器，降低运维成本
- 按需付费，成本优化
- 自动扩缩容，应对突发流量
- 快速迭代，专注于业务逻辑

**限制**：
- 冷启动延迟（cold start）
- 供应商锁定（vendor lock-in）
- 调试和监控复杂
- 不适合长时间运行的任务

## 答题技巧与写作指导

### 阅读题目的方法

**第一遍：快速浏览（1~2 分钟）**
- 了解题目整体结构和业务场景
- 明确题目要求和分值分布
- 标记关键信息和专业术语

**第二遍：精读问题（2~3 分钟）**
- 仔细阅读每个小问的要求
- 理解问题的考查重点
- 预估每个问题的答题深度

**第三遍：带着问题读案例（5~8 分钟）**
- 针对每个问题在案例中寻找线索
- 标记与问题相关的段落
- 记录关键数据和事实

### 分析问题的框架

**STAR 分析框架**：
- **Situation**（情境）：当前系统的状况
- **Task**（任务）：需要解决的问题
- **Action**（行动）：可采取的措施
- **Result**（结果）：预期效果

**SWOT 分析框架**：
- **Strengths**（优势）：现有系统的优点
- **Weaknesses**（劣势）：现有系统的不足
- **Opportunities**（机会）：外部环境带来的机遇
- **Threats**（威胁）：外部环境的挑战

### 组织答案的结构

**1. 总-分-总结构**
- 开头：简要概括分析结论
- 中间：分点详细阐述，每点有小标题
- 结尾：总结建议或预期效果

**2. 问题-分析-解决结构**
- 明确指出问题
- 深入分析问题根源
- 提出解决方案并说明理由

**3. 比较-选择-实施结构**
- 列举可选方案
- 比较各方案的优劣
- 给出推荐方案和实施步骤

### 时间分配建议

| 阶段 | 时间 | 内容 |
|------|------|------|
| 阅读题目 | 10 分钟 | 通读案例和问题，标记重点 |
| 构思答案 | 15 分钟 | 针对每道题构思答题要点 |
| 撰写答案 | 55 分钟 | 书写四道题的答案（平均 13 分钟/题） |
| 检查完善 | 10 分钟 | 补充遗漏点，检查语法和格式 |

::: warning 重点难点

**难点一：架构图的绘制**
- 建议使用分层方式描述，从上到下：用户界面 → 接入层 → 服务层 → 数据层
- 标注关键组件和数据流向
- 用文字说明图中无法完全表达的信息

**难点二：质量属性的量化分析**
- 能够将定性分析转化为定量指标（如响应时间从 5s 降到 500ms）
- 熟悉 SLA（服务等级协议）指标（如 99.9% 可用性 = 年 downtime < 9 小时）
- 能够评估改进方案的实际效果和成本

**难点三：新技术应用的权衡**
- 每种新技术都有适用场景和代价
- 要能够根据业务特点做出合理选择
- 关注技术的成熟度和社区生态

:::

## 历年真题解析示例

### 示例一：微服务架构拆分设计

#### 题目

某电商公司原有的单体应用已经无法满足业务发展需要。该系统包含以下功能模块：
- 用户管理：注册、登录、个人信息管理
- 商品管理：商品展示、分类管理、库存管理
- 订单管理：订单创建、支付、配送跟踪
- 营销管理：促销活动、优惠券、积分
- 数据统计：销售报表、用户行为分析

当前系统的问题：
1. 代码库庞大（约 200 万行代码），维护困难
2. 发布周期长（每月一次），无法快速响应市场变化
3. 某个模块的问题可能导致整个系统崩溃
4. 技术栈统一，无法使用最适合的技术

**问题**：
1. 请设计该系统的微服务拆分方案，说明拆分原则和每个服务的职责。（12 分）
2. 请设计服务间的通信方式，并说明如何保障通信的可靠性。（8 分）

#### 分析过程

**1. 服务拆分原则**
- **单一职责原则（SRP）**：每个服务只负责一个业务领域
- **领域驱动设计（DDD）**：按限界上下文（Bounded Context）拆分
- **团队自治**：每个服务可由独立团队负责
- **数据独立**：每个服务拥有自己的数据库
- **渐进式拆分**：从核心模块开始，逐步迁移

**2. 服务划分**

根据业务领域，建议拆分为以下微服务：

| 服务名称 | 职责 | 核心功能 |
|----------|------|----------|
| 用户服务 | 用户生命周期管理 | 注册、登录、权限、个人信息 |
| 商品服务 | 商品信息管理 | 商品展示、分类、搜索、库存 |
| 订单服务 | 交易流程管理 | 订单创建、支付、配送、退款 |
| 营销服务 | 促销活动管理 | 优惠券、满减、积分、秒杀 |
| 统计服务 | 数据分析与报表 | 销售统计、用户行为分析 |

**3. 通信方式设计**

- **同步通信（RESTful API）**：适用于需要即时响应的场景
  - 用户查询商品信息
  - 创建订单时验证库存
  - 支付回调通知

- **异步通信（消息队列）**：适用于最终一致性场景
  - 订单创建后发送通知（短信、邮件）
  - 用户行为数据收集
  - 库存扣减后的数据同步

**4. 可靠性保障措施**
- **服务熔断**：使用熔断器（Circuit Breaker）防止级联故障
- **重试机制**：对临时性故障进行重试（需设置重试次数和退避策略）
- **超时控制**：设置合理的超时时间，避免请求堆积
- **限流降级**：高峰期对非核心服务进行降级
- **服务注册与发现**：使用 Nacos/Eureka/Consul 实现服务自动发现
- **健康检查**：定期检查服务健康状态，自动剔除异常实例

#### 参考答案

**（1）微服务拆分方案**

根据领域驱动设计原则，按业务边界将系统拆分为 5 个核心微服务：

1. **用户服务（User Service）**
   - 职责：管理用户账户、认证授权、个人信息
   - 数据库：用户数据库（用户表、角色表、权限表）
   - API：登录、注册、信息查询、密码修改

2. **商品服务（Product Service）**
   - 职责：管理商品信息、分类、库存、搜索
   - 数据库：商品数据库（商品表、分类表、库存表）
   - API：商品查询、库存管理、分类管理

3. **订单服务（Order Service）**
   - 职责：管理订单生命周期、支付、配送
   - 数据库：订单数据库（订单表、订单项表、支付记录表）
   - API：创建订单、查询订单、取消订单

4. **营销服务（Marketing Service）**
   - 职责：管理促销活动、优惠券、积分
   - 数据库：营销数据库（活动表、优惠券表、积分表）
   - API：活动查询、领券、积分计算

5. **统计服务（Analytics Service）**
   - 职责：收集业务数据、生成报表、用户画像
   - 数据库：数据仓库（事实表、维度表）
   - API：报表查询、数据导出

**拆分原则**：
- 按业务领域拆分，每个服务对应一个限界上下文
- 服务间通过 API 契约交互，禁止直接访问其他服务的数据库
- 优先拆分核心业务模块，非核心模块可逐步迁移

**（2）服务通信与可靠性保障**

**通信方式选择**：
- **同步通信（RESTful API）**：用于需要即时响应的场景，如查询商品、创建订单
- **异步通信（消息队列 Kafka/RabbitMQ）**：用于不需要即时响应的场景，如发送通知、数据同步

**可靠性保障机制**：

| 机制 | 实现方案 | 作用 |
|------|----------|------|
| 服务熔断 | Hystrix/Resilience4j | 防止故障扩散 |
| 重试机制 | Spring Retry + 退避策略 | 处理临时性故障 |
| 超时控制 | Feign/Ribbon 配置 | 避免请求堆积 |
| 限流降级 | Sentinel/Resilience4j | 保护系统资源 |
| 服务发现 | Nacos/Eureka | 动态服务定位 |
| 健康检查 | /actuator/health | 自动故障转移 |

### 示例二：系统可用性改进设计

#### 题目

某银行的核心支付系统当前为单机部署，架构如下：
- 1 台应用服务器（Tomcat）
- 1 台数据库服务器（MySQL）
- 前端通过负载均衡器（SLB）接入

该系统在过去的 12 个月中发生了 3 次故障，每次故障时间约为 2~4 小时，造成以下影响：
1. 用户无法进行支付操作
2. 交易数据丢失（未持久化到数据库）
3. 银行声誉受损

**问题**：
1. 分析当前架构存在的单点故障问题。（5 分）
2. 请设计一个高可用架构方案，要求系统可用性达到 99.99%。（15 分）
3. 请说明如何保障数据的一致性。（5 分）

#### 分析过程

**1. 单点故障分析**
- 应用服务器：单台 Tomcat，故障后整个应用不可用
- 数据库服务器：单台 MySQL，故障后数据丢失和服务中断
- SLB：虽然是负载均衡器，但如果是单台部署也有单点问题

**2. 高可用架构设计思路**
- 消除单点：所有组件都需要冗余部署
- 自动故障转移：某个节点故障时，流量自动切换到备用节点
- 数据持久化：确保交易数据不丢失
- 多活部署：考虑跨可用区甚至跨地域部署

**3. 数据一致性保障**
- 事务管理：ACID 特性保障
- 数据备份：定期全量备份 + 实时增量备份
- 数据同步：主从复制、读写分离
- 最终一致性：通过消息队列保障最终一致

#### 参考答案

**（1）单点故障分析**

当前架构存在以下单点故障：

1. **应用服务器单点**：单台 Tomcat 一旦宕机，整个支付服务不可用
2. **数据库服务器单点**：单台 MySQL 一旦宕机，不仅服务中断，还可能造成数据丢失
3. **SLB 单点**：虽然 SLB 本身高可用，但如果是单台部署，也存在风险

**故障影响量化**：
- 年故障时间：3 次 × 3 小时（平均）= 9 小时
- 年可用时间：8760 - 9 = 8751 小时
- 当前可用性：8751 / 8760 ≈ 99.90%

**（2）高可用架构方案**

目标可用性：99.99%
- 年允许故障时间：8760 × 0.01% = 0.876 小时 ≈ 53 分钟
- 月允许故障时间：730 × 0.01% ≈ 5 分钟

**整体架构**：

**各层高可用设计**：

1. **接入层**
   - DNS 轮询 + GSLB（全局负载均衡）
   - SLB 采用主备模式或集群模式
   - 多地域接入，支持异地容灾

2. **应用层**
   - 至少 3 台应用服务器，部署在不同可用区（AZ）
   - 无状态设计，支持水平扩展
   - 健康检查 + 自动故障转移

3. **服务层**
   - 微服务架构，每个服务至少 2 实例
   - 服务注册与发现（Nacos/Eureka）
   - 熔断、降级、限流保护

4. **数据层**
   - MySQL 主从复制（1 主 2 从）
   - 半同步复制（semi-sync）保障数据不丢失
   - 自动故障转移（MHA/Orchestrator）
   - 定期全量备份 + 实时增量备份（binlog）

5. **缓存层**
   - Redis 集群模式（3 主 3 从）
   - 持久化策略（AOF + RDB）
   - 缓存穿透、击穿、雪崩防护

**（3）数据一致性保障**

| 机制 | 方案 | 效果 |
|------|------|------|
| 数据库事务 | ACID 特性 | 强一致性保障 |
| 主从复制 | 半同步复制 | 数据实时同步到从库 |
| 数据备份 | 每日全量 + 实时 binlog | 可恢复到任意时间点 |
| 最终一致性 | 消息队列 + 补偿机制 | 分布式场景下数据最终一致 |
| 数据校验 | 定期数据对账 | 及时发现数据不一致 |

**改进后可用性评估**：
- 应用层：3 节点，单节点故障不影响服务，可用性 99.9%
- 数据层：主从 + 自动切换，故障恢复时间 < 30 秒，可用性 99.99%
- 整体可用性：> 99.99%（满足目标）
