# 第四章：软件工程

> 分值占比：10%-15% | 重要程度：★★★★★

## 考情分析

软件工程是系统架构设计师考试的重点章节，分值占比高。本章涵盖软件生命周期模型、需求工程、系统分析与设计、软件测试、净室软件工程、基于构件的软件工程等内容。

---

## 4.1 软件生命周期模型

### 4.1.1 瀑布模型

- **阶段**：需求分析 → 概要设计 → 详细设计 → 编码实现 → 测试 → 运行维护
- **特点**：阶段严格顺序，阶段间有文档交付
- **优点**：结构清晰，易于管理
- **缺点**：需求变化响应慢，后期修改代价大
- **适用**：需求明确、稳定的项目

### 4.1.2 迭代模型

- **特点**：将开发过程分为多个迭代周期
- **每次迭代**：包含需求、设计、实现、测试
- **优点**：逐步完善，风险分散
- **缺点**：管理复杂度高
- **适用**：需求不明确的大型项目

### 4.1.3 敏捷模型

- **特点**：迭代增量、快速响应变化
- **敏捷宣言**：
  - 个体和互动高于流程和工具
  - 工作的软件高于详尽的文档
  - 客户合作高于合同谈判
  - 响应变化高于遵循计划
- **典型方法**：Scrum、XP（极限编程）、看板
- **适用**：需求变化快、创新性项目

### 4.1.4 螺旋模型

- **特点**：风险驱动，结合瀑布与迭代
- **四个象限**：制定计划、风险分析、实施工程、客户评估
- **优点**：风险早期识别和处理
- **缺点**：风险管理成本高
- **适用**：高风险、大型复杂项目

## 4.2 需求工程

### 4.2.1 需求分类

- **功能需求**：系统应提供的功能
- **非功能需求**：性能、可靠性、安全性、可用性等
- **业务规则**：业务约束、法律合规

### 4.2.2 需求获取技术

- 用户访谈
- 问卷调查
- 现场观察
- 原型法
- 用例分析

### 4.2.3 需求分析方法

- **结构化分析（SA）**：数据流图（DFD）、数据字典
- **面向对象分析（OOA）**：用例图、类图、序列图
- **原型分析**：快速构建原型，迭代确认需求

### 4.2.4 需求管理

- 需求跟踪：正向跟踪（需求→设计）、反向跟踪（设计→需求）
- 需求变更控制：变更申请、影响分析、批准实施
- 需求优先级：MoSCoW方法（Must、Should、Could、Won't）

## 4.3 系统分析与设计

### 4.3.1 结构化设计

- **概要设计**：模块划分、接口定义、数据结构设计
- **详细设计**：算法设计、接口细节、数据结构实现
- **设计原则**：模块独立性（高内聚、低耦合）

### 4.3.2 面向对象设计

- **类与对象**：类是对对象的抽象，对象是类的实例
- **封装**：隐藏内部实现，暴露接口
- **继承**：子类继承父类的属性和方法
- **多态**：同一操作作用于不同对象产生不同结果
- **关联**：类之间的关系（关联、聚合、组合）

### 4.3.3 设计模式

- **创建型模式**：单例、工厂方法、抽象工厂、建造者
- **结构型模式**：适配器、装饰器、代理、外观、组合
- **行为型模式**：观察者、策略、命令、迭代器、模板方法

## 4.4 软件测试

### 4.4.1 测试级别

- **单元测试**：测试单个模块或函数
- **集成测试**：测试模块间的接口和交互
- **系统测试**：测试完整系统
- **验收测试**：用户确认系统满足需求

### 4.4.2 测试方法

- **白盒测试**：基于代码结构，覆盖语句、分支、路径
- **黑盒测试**：基于功能，不关心内部结构
  - 等价类划分
  - 边界值分析
  - 因果图法
  - 错误推测法

### 4.4.3 测试类型

- 功能测试、性能测试、安全测试、兼容性测试
- 回归测试、压力测试、负载测试

### 4.4.4 测试用例设计

- 测试用例 = 输入数据 + 预期输出 + 测试步骤
- 用例覆盖率应达到合理标准（建议≥80%）

## 4.5 净室软件工程

### 4.5.1 净室方法概述

净室软件工程（Cleanroom Software Engineering）强调正确性验证和统计测试，追求零缺陷软件。

### 4.5.2 净室技术

- **盒结构规约**：黑盒、状态盒、清晰盒
- **正确性验证**：形式化方法证明程序正确性
- **统计测试**：基于使用模型的统计测试
- **增量开发**：增量式开发与集成

### 4.5.3 净室特点

- 不依赖调试，强调预防缺陷
- 基于数学理论验证正确性
- 软件认证：质量评估与发布决策

## 4.6 基于构件的软件工程（CBSE）

### 4.6.1 CBSE定义

基于构件的软件工程（Component-Based Software Engineering）通过复用已有构件来构建软件系统。

### 4.6.2 构件特性

- **可复用性**：可在多个系统中使用
- **可替换性**：可被功能类似的构件替换
- **标准化接口**：提供规范的接口规范
- **独立部署**：可独立发布和部署

### 4.6.3 构件分类

- **系统级构件**：操作系统、数据库、中间件
- **业务级构件**：业务逻辑组件
- **专用构件**：特定领域构件

### 4.6.4 CBSE优势

- 提高开发效率
- 保证软件质量
- 降低维护成本
- 支持快速原型

## 4.7 软件项目管理与CMMI

### 4.7.1 软件项目管理要素

- **范围管理**：定义和控制项目范围
- **进度管理**：制定计划、跟踪进度
- **成本管理**：预算编制、成本控制
- **质量管理**：质量计划、质量控制、质量保证
- **风险管理**：风险识别、评估、应对
- **配置管理**：版本控制、变更管理

### 4.7.2 CMMI模型

CMMI（Capability Maturity Model Integration）是能力成熟度模型集成。

**五个成熟度级别：**
- **级别1（初始级）**：过程无序，成功依赖个人
- **级别2（已管理级）**：项目级过程管理
- **级别3（已定义级）**：组织级标准化过程
- **级别4（量化管理级）**：过程量化度量
- **级别5（优化级）**：持续过程改进

### 4.7.3 CMMI过程域

- **项目管理类**：项目计划、监控、集成管理
- **工程类**：需求开发、技术方案、产品集成
- **支持类**：配置管理、质量保证、测量分析

## 考点速记

::: tip 高频考点
1. 四种生命周期模型特点与适用场景（瀑布、迭代、敏捷、螺旋）
2. 需求工程过程：获取、分析、规格说明、验证、管理
3. 结构化分析工具：数据流图（DFD）、数据字典
4. 软件测试级别：单元、集成、系统、验收
5. 白盒测试覆盖标准：语句覆盖、分支覆盖、路径覆盖
6. 黑盒测试方法：等价类划分、边界值分析
7. 设计模式分类：创建型、结构型、行为型
8. CMMI五个级别及核心特征
:::

::: warning 重点难点
- **瀑布模型 vs 敏捷模型**：瀑布适用于需求稳定项目，敏捷适用于需求变化快的项目
- **需求变更管理**：必须经过正式变更控制流程
- **高内聚低耦合**：模块独立性的核心原则
- **测试用例设计**：白盒测试关注代码结构，黑盒测试关注功能
- **CMMI级别特征**：从无序到有序，从定性到定量，从被动到主动改进
:::
