# 第七章：分布式系统设计

> 分值占比：10%-15% | 重要程度：★★★★★

## 考情分析

分布式系统设计是系统架构设计师考试的重点章节，主要考查分布式系统的基本概念、CAP/BASE理论、分布式事务、分布式缓存、消息队列、分布式共识算法等。考试重点在于分布式系统的透明性、CAP理论的权衡、分布式事务的解决方案、缓存策略与一致性问题，以及主流分布式中间件的选型。案例分析题常涉及分布式架构设计、数据一致性方案、缓存策略等。

---

## 7.1 分布式系统基础

### 7.1.1 分布式系统定义与特征

- **定义**：由多台独立计算机通过网络互联，协同完成共同任务的系统
- **主要特征**：
  - 并发性：多个进程同时运行
  - 缺乏全局时钟：各节点时间不一致
  - 故障独立性：各节点可能独立故障

### 7.1.2 分布式系统透明性

- **访问透明性**：隐藏数据表示和访问方式的差异
- **位置透明性**：隐藏服务的位置信息
- **迁移透明性**：隐藏服务是否迁移
- **重定位透明性**：隐藏服务是否重定位
- **故障透明性**：隐藏故障和恢复
- **性能透明性**：隐藏性能变化
- **伸缩透明性**：隐藏系统规模扩展

### 7.1.3 分布式系统设计问题

- **进程间通信**：消息传递、远程过程调用（RPC）
- **命名服务**：DNS、Naming Service
- **同步问题**：时钟同步、事件排序
- **一致性与复制**：数据副本管理
- **容错性**：故障检测与恢复
- **安全性**：认证、授权、加密

## 7.2 CAP理论与BASE理论

### 7.2.1 CAP定理

**CAP定理**：分布式系统不可能同时满足以下三个特性：

| 特性 | 描述 |
|------|------|
| 一致性（Consistency） | 所有节点在同一时刻看到相同的数据 |
| 可用性（Availability） | 每个请求都能得到响应（不保证最新） |
| 分区容错性（Partition Tolerance） | 系统在网络分区时仍能工作 |

**核心结论：**
- P是必须满足的（网络故障不可避免）
- 实际系统在C和A之间权衡
- CP系统：优先保证一致性，可用性可能下降（如ZooKeeper）
- AP系统：优先保证可用性，允许临时不一致（如Cassandra）

### 7.2.2 BASE理论

**BASE理论**是CAP定理的延伸，强调基本可用、软状态和最终一致性：

- **基本可用（Basically Available）**
  - 系统在部分故障时仍能提供服务
  - 损失部分性能或功能，但核心功能可用

- **软状态（Soft State）**
  - 允许系统存在中间状态
  - 副本间同步存在延迟

- **最终一致性（Eventually Consistent）**
  - 经过一段时间后，所有副本最终一致
  - 不同场景下最终一致性的时间不同

## 7.3 分布式事务

### 7.3.1 分布式事务问题

- **场景**：跨多个服务或数据库的操作需要原子性
- **挑战**：网络故障、节点故障、数据一致性问题

### 7.3.2 两阶段提交（2PC）

**协调者**：管理事务提交过程

**阶段一：准备阶段**
- 协调者询问所有参与者是否可以提交
- 参与者执行事务，记录日志，但不提交
- 参与者返回"同意"或"中止"

**阶段二：提交阶段**
- 若所有参与者同意，协调者发送提交命令
- 若任一参与者中止，协调者发送回滚命令

**优缺点：**
- 优点：强一致性，保证原子性
- 缺点：
  - 同步阻塞：所有参与者等待协调者
  - 单点故障：协调者故障导致系统挂起
  - 数据不一致：协调者发送部分提交后崩溃

### 7.3.3 三阶段提交（3PC）

**阶段一：CanCommit**（询问阶段）
- 协调者询问参与者是否可以执行事务
- 参与者不执行，仅检查状态

**阶段二：PreCommit**（预提交阶段）
- 若都同意，协调者发送预提交命令
- 参与者执行事务，锁定资源

**阶段三：DoCommit**（提交阶段）
- 协调者发送提交命令
- 参与者正式提交

**改进：**
- 引入超时机制，减少阻塞
- 第一阶段不执行事务，降低风险

### 7.3.4 TCC（Try-Confirm-Cancel）

- **Try阶段**：资源检查与预留
- **Confirm阶段**：确认执行，真正提交
- **Cancel阶段**：取消执行，释放预留资源

**特点：**
- 应用层实现，灵活性高
- 最终一致性
- 需要业务实现幂等性

### 7.3.5 可靠消息最终一致性

- 通过消息队列实现事务消息
- 本地事务与消息发送原子性
- 下游消费实现最终一致

### 7.3.6 最大努力通知

- 事务发起方执行事务后发送通知
- 事务参与方最大努力消费，不保证成功
- 配合补偿机制

## 7.4 分布式缓存

### 7.4.1 缓存概述

- **定义**：将热点数据存储在离用户更近的存储层
- **目的**：降低数据库压力、提高响应速度

### 7.4.2 缓存策略

- **Cache Aside（旁路缓存）**
  - 读：先读缓存，未命中则读数据库，再写缓存
  - 写：先写数据库，再删除缓存
  - 最常用，适合一致性要求不高的场景

- **Read/Write Through**
  - 应用只与缓存交互，缓存负责与数据库同步
  - 读穿透：缓存未命中时自动读数据库
  - 写穿透：写缓存时同时写数据库

- **Write Behind（异步写回）**
  - 写缓存即返回，异步写数据库
  - 性能最好，但一致性最差

### 7.4.3 缓存问题

- **缓存穿透**：查询不存在的数据，请求直达数据库
  - 解决：缓存空值、布隆过滤器

- **缓存击穿**：热点key过期，大量请求打到数据库
  - 解决：互斥锁、热点key不过期

- **缓存雪崩**：大量key同时过期或缓存服务宕机
  - 解决：过期时间加随机值、集群部署

### 7.4.4 缓存一致性

- **强一致性**：缓存与数据库实时同步（性能差）
- **最终一致性**：允许短暂不一致，通过策略恢复
- **一致性策略**：
  - 更新数据库后删除缓存（Cache Aside）
  - 设置合理的过期时间
  - 消息队列异步更新缓存

### 7.4.5 一致性哈希

- **目的**：解决分布式缓存节点增减时的数据迁移问题
- **原理**：将节点和数据映射到哈希环
- **优点**：节点增减时只影响相邻节点，数据迁移量小
- **虚拟节点**：解决数据倾斜问题

## 7.5 消息队列

### 7.5.1 消息队列概述

- **定义**：分布式系统中用于异步通信的中间件
- **核心模式**：
  - 点对点（P2P）：一对一通信，消息被消费后删除
  - 发布/订阅（Pub/Sub）：一对多通信，消息被所有订阅者消费

### 7.5.2 主流消息队列对比

| 消息队列 | 特点 | 适用场景 |
|---------|------|---------|
| RabbitMQ | 功能完整，支持多种协议 | 企业级消息通信 |
| Kafka | 高吞吐、持久化、流处理 | 日志收集、流处理、事件驱动 |
| RocketMQ | 高可靠、事务消息 | 金融级消息、订单场景 |
| ActiveMQ | 成熟稳定 | 传统企业应用 |

### 7.5.3 消息队列应用场景

- **异步处理**：用户注册后发送邮件/短信
- **应用解耦**：订单系统与库存系统解耦
- **流量削峰**：秒杀、抢购场景的流量控制
- **日志收集**：应用日志的集中处理

### 7.5.4 消息可靠性

- **生产可靠性**：确认机制（ACK）
- **存储可靠性**：持久化、副本机制
- **消费可靠性**：ACK机制、重试策略

### 7.5.5 消息幂等性

- **问题**：网络重传导致消息重复消费
- **解决**：唯一消息ID + 去重表、业务幂等设计

## 7.6 分布式共识算法

### 7.6.1 Paxos算法

- **提出者**：Leslie Lamport
- **核心问题**：分布式系统中如何就某个值达成一致
- **角色**：
  - Proposer：提出提案
  - Acceptor：接受或拒绝提案
  - Learner：学习被选定的值

- **流程**：
  - Prepare阶段：Proposer发送提案编号，Acceptor承诺不接受更小编号
  - Accept阶段：Proposer发送提案值，多数Acceptor接受则通过

- **缺点**：难以理解，工程实现复杂

### 7.6.2 Raft算法

- **目的**：比Paxos更易理解和实现
- **核心思想**：强领导者模型

**角色：**
- **Leader**：处理所有客户端请求，复制日志
- **Follower**：被动响应Leader和Candidate
- **Candidate**：选举新Leader时的临时角色

**选举流程：**
1. Follower超时未收到Leader心跳，转为Candidate
2. 发起选举，投票给自己
3. 获得多数票成为Leader
4. 定期发送心跳维持权威

**日志复制：**
1. Leader接收客户端请求，写入本地日志
2. Leader复制日志到所有Follower
3. 多数Follower确认后，Leader提交日志
4. Leader通知Follower提交日志

**特点：**
- 易于理解和实现
- 强一致性
- 广泛用于工业系统（etcd、Consul、TiKV）

### 7.6.3 ZAB协议

- **应用**：ZooKeeper
- **特点**：原子广播、崩溃恢复
- **两种模式**：广播模式（正常运行）、恢复模式（Leader故障）

## 考点速记

::: tip 高频考点
1. CAP定理：P必须满足，C和A之间权衡
2. CP系统：ZooKeeper、etcd；AP系统：Cassandra、Eureka
3. BASE理论：基本可用、软状态、最终一致性
4. 分布式事务：2PC（强一致，同步阻塞）、3PC（超时机制）、TCC（应用层）
5. 缓存策略：Cache Aside最常用；Write Behind性能最好
6. 缓存问题：穿透（空值/布隆）、击穿（互斥锁）、雪崩（随机过期）
7. 消息队列：Kafka高吞吐、RocketMQ高可靠、RabbitMQ功能全
8. 共识算法：Paxos理论基础，Raft工程友好，ZAB用于ZooKeeper
9. 一致性哈希：解决分布式缓存节点增减问题
10. 微服务与SOA：微服务更轻量、去中心化，SOA更重、企业级集成
:::

## 重点难点

::: warning 重点难点
1. **CAP权衡**：P是前提，实际系统根据业务选择CP或AP；金融系统倾向CP，互联网系统倾向AP
2. **2PC问题**：同步阻塞、单点故障、数据不一致；3PC改进超时机制但不能完全解决
3. **缓存一致性**：Cache Aside（先更数据库再删缓存）最常用；需处理并发更新导致脏数据
4. **消息队列可靠性**：生产端ACK、存储端持久化、消费端幂等性三者缺一不可
5. **Raft选举**：Leader需获得多数票；同一Term最多一个Leader；日志连续性保证
6. **分布式事务选择**：2PC适合强一致场景；TCC适合业务可控场景；可靠消息适合最终一致场景
:::
