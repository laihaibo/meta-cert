# 第九章：大数据架构

> 分值占比：5%-10% | 重要程度：★★★

## 考情分析

大数据架构是系统架构设计师考试的新增内容，主要考查大数据基本概念、Hadoop生态系统、大数据存储与处理框架、数据仓库与数据湖等。考试重点在于大数据的4V特征、HDFS存储原理、MapReduce计算模型、Spark内存计算、NoSQL数据库类型及适用场景。综合知识题常涉及概念辨析，案例分析题可能涉及大数据架构选型和设计。

---

## 9.1 大数据基础

### 9.1.1 大数据定义与特征

- **定义**：无法用传统数据处理工具在合理时间内处理的海量、高增长、多样化的信息资产
- **4V特征**：
  - **Volume（大量）**：数据规模巨大，从TB到PB级
  - **Velocity（高速）**：数据产生和处理速度快
  - **Variety（多样）**：结构化、半结构化、非结构化数据
  - **Value（价值）**：价值密度低，需要深度挖掘

### 9.1.2 大数据技术生态

- **数据采集**：Flume、Sqoop、Logstash
- **数据存储**：HDFS、HBase、NoSQL
- **数据处理**：MapReduce、Spark、Flink
- **数据分析**：Hive、Pig、Spark SQL
- **数据可视化**：Tableau、Power BI、ECharts

### 9.1.3 大数据应用场景

- 互联网搜索：搜索引擎索引
- 推荐系统：用户行为分析
- 金融风控：交易数据分析
- 智慧城市：物联网数据融合
- 医疗健康：病历数据分析

## 9.2 Hadoop生态系统

### 9.2.1 Hadoop概述

- **定义**：分布式计算框架，解决海量数据存储和计算问题
- **核心组成**：
  - **HDFS**：分布式文件系统
  - **MapReduce**：分布式计算框架
  - **YARN**：资源管理和调度框架

### 9.2.2 HDFS（Hadoop Distributed File System）

- **设计目标**：存储海量文件，高容错、高吞吐
- **核心组件**：
  - **NameNode**：主节点，管理文件系统的命名空间和元数据
  - **DataNode**：从节点，存储实际数据块
  - **Secondary NameNode**：辅助NameNode，定期合并镜像

- **数据块（Block）**：
  - 默认大小：128MB（Hadoop 2.x+）
  - 最后一个Block可能不完整
  - 每个Block默认3副本

- **数据写入流程**：
  1. 客户端请求NameNode写入
  2. NameNode返回DataNode地址
  3. 客户端写入第一个DataNode
  4. 流水线复制到其他DataNode
  5. 写入完成通知NameNode

- **数据读取流程**：
  1. 客户端请求NameNode获取Block位置
  2. 选择最近DataNode读取
  3. 关闭连接

- **HDFS特点**：
  - 高容错：副本机制、自动恢复
  - 高吞吐：适合批量读写
  - 不适合低延迟随机访问

### 9.2.3 MapReduce

- **核心思想**：分而治之（Divide and Conquer）
- **两个阶段**：
  - **Map阶段**：输入分片，并行处理，输出键值对
  - **Reduce阶段**：按键分组，聚合计算，输出结果

- **执行流程**：
  1. 输入分片（Input Split）
  2. Map任务执行
  3. Shuffle阶段（分区、排序、合并）
  4. Reduce任务执行
  5. 输出结果

- **优缺点**：
  - 优点：高容错、高扩展、适合离线批处理
  - 缺点：延迟高（分钟级）、不适合实时计算、磁盘IO开销大

### 9.2.4 YARN

- **定义**：Yet Another Resource Negotiator，Hadoop资源管理器
- **核心组件**：
  - **ResourceManager**：全局资源管理器
  - **NodeManager**：单节点资源管理器
  - **ApplicationMaster**：每个应用的管理者

- **YARN vs 传统MapReduce**：
  - 解耦资源管理与计算框架
  - 支持多种计算框架（MapReduce、Spark、Storm）
  - 更好的资源利用率

## 9.3 大数据存储

### 9.3.1 NoSQL数据库

- **出现原因**：传统关系型数据库无法满足大数据场景的扩展性需求
- **四大类型**：

| 类型 | 代表产品 | 数据模型 | 适用场景 |
|------|---------|---------|---------|
| 键值存储（Key-Value） | Redis、DynamoDB | 键值对 | 缓存、会话存储 |
| 文档存储（Document） | MongoDB、CouchDB | JSON/BSON文档 | 内容管理、用户配置 |
| 列族存储（Column Family） | HBase、Cassandra | 列族 | 海量数据、时序数据 |
| 图数据库（Graph） | Neo4j、JanusGraph | 图结构 | 社交网络、推荐系统 |

### 9.3.2 HBase

- **定义**：分布式、面向列的NoSQL数据库，构建于HDFS之上
- **数据模型**：
  - 表（Table）
  - 行键（Row Key）：主键，字典序排序
  - 列族（Column Family）：列的分组，物理存储的基本单位
  - 列限定符（Column Qualifier）：列族内的具体列
  - 时间戳（Timestamp）：版本控制

- **核心特点**：
  - 稀疏：空列不占用存储
  - 分布式：数据分布存储在多个RegionServer
  - 高性能：基于内存和磁盘混合存储

- **架构组件**：
  - **HMaster**：管理元数据、Region分配
  - **RegionServer**：管理Region，处理读写请求
  - **ZooKeeper**：协调服务，故障转移

### 9.3.3 NewSQL数据库

- **定义**：结合NoSQL扩展性和关系型数据库ACID特性的新一代数据库
- **代表产品**：Spanner、TiDB、CockroachDB
- **特点**：
  - 水平扩展
  - 强一致性
  - SQL支持

## 9.4 大数据处理框架

### 9.4.1 Spark

- **定义**：基于内存的分布式计算框架
- **核心特性**：
  - 内存计算：比MapReduce快10-100倍
  - 多语言支持：Scala、Java、Python、R
  - 统一引擎：批处理、流处理、SQL、机器学习

- **核心组件**：
  - **Spark Core**：基础计算引擎，RDD抽象
  - **Spark SQL**：结构化数据处理
  - **Spark Streaming**：流处理（微批）
  - **MLlib**：机器学习库
  - **GraphX**：图计算

- **RDD（弹性分布式数据集）**：
  - 不可变、可分区、可并行计算
  - 血缘关系：Lineage，容错恢复基础
  - 转换操作（Transformation）：map、filter、flatMap
  - 行动操作（Action）：count、collect、save

- **Spark vs MapReduce**：

| 对比项 | MapReduce | Spark |
|--------|-----------|-------|
| 计算模型 | 磁盘IO | 内存计算 |
| 执行模式 | 批处理 | 批处理+流处理 |
| 速度 | 慢 | 快 |
| 迭代计算 | 差（多次磁盘IO） | 好（内存缓存） |
| 适用场景 | 离线批处理 | 离线+实时+机器学习 |

### 9.4.2 Flink

- **定义**：流批一体的分布式计算框架
- **核心特点**：
  - 真正的流处理：事件驱动，逐条处理
  - 低延迟：毫秒级
  - 高吞吐：每秒处理百万级事件
  -  Exactly-Once语义：精确一次处理保证

- **核心组件**：
  - **JobManager**：任务调度、协调
  - **TaskManager**：执行任务
  - **DataStream API**：流处理API
  - **Table API & SQL**：结构化流处理

- **Flink vs Spark Streaming**：

| 对比项 | Spark Streaming | Flink |
|--------|----------------|-------|
| 处理模型 | 微批处理 | 真正的流处理 |
| 延迟 | 秒级 | 毫秒级 |
| 窗口支持 | 有限 | 丰富（滚动、滑动、会话、会话窗口） |
| 状态管理 | 有限 | 强大（状态后端、检查点） |

### 9.4.3 Kafka

- **定义**：高吞吐分布式消息队列，流处理平台
- **核心概念**：
  - **Topic**：消息类别
  - **Partition**：Topic的分区，并行处理单元
  - **Broker**：Kafka服务器节点
  - **Producer**：消息生产者
  - **Consumer Group**：消费者组，组内消费者分担消费

- **特点**：
  - 高吞吐：每秒百万级消息
  - 持久化：消息持久化到磁盘
  - 分布式：支持集群部署
  - 流处理：Kafka Streams、KSQL

## 9.5 数据仓库与数据湖

### 9.5.1 数据仓库

- **定义**：面向主题的、集成的、相对稳定的、反映历史变化的数据集合
- **特点**：
  - 面向主题：围绕业务主题组织数据
  - 集成性：数据来自多个异构源
  - 稳定性：主要进行查询分析，不常修改
  - 历史性：保存历史数据

- **数据仓库分层**：
  - **ODS层**：原始数据层，与源系统结构一致
  - **DWD层**：明细数据层，清洗和标准化
  - **DWS层**：汇总数据层，按主题聚合
  - **ADS层**：应用数据层，面向具体应用

### 9.5.2 数据湖

- **定义**：存储原始数据（结构化、半结构化、非结构化）的大规模存储库
- **特点**：
  - 存储原始数据，不预先定义模式
  - 支持schema-on-read（读时定义模式）
  - 适合数据探索和机器学习

- **数据湖 vs 数据仓库**：

| 对比项 | 数据仓库 | 数据湖 |
|--------|---------|--------|
| 数据类型 | 结构化 | 结构化+半结构化+非结构化 |
| Schema | Schema-on-write（写时定义） | Schema-on-read（读时定义） |
| 处理方式 | ETL（抽取转换加载） | ELT（抽取加载转换） |
| 用户 | 业务分析师 | 数据科学家 |
| 成本 | 高 | 低 |

### 9.5.3 Lambda架构与Kappa架构

**Lambda架构**：
- **批处理层**：处理全量数据，保证准确性
- **速度层**：处理实时数据，保证低延迟
- **服务层**：合并批处理和实时结果

**Kappa架构**：
- 简化Lambda，只保留流处理层
- 所有数据通过流处理处理
- 历史数据重放处理

## 9.6 大数据架构设计

### 9.6.1 大数据架构模式

- **批处理架构**：Hadoop MapReduce、Spark
- **流处理架构**：Flink、Storm
- **Lambda架构**：批处理+流处理结合
- **Kappa架构**：纯流处理

### 9.6.2 数据一致性

- **Exactly-Once**：精确一次处理（Flink checkpoint、Kafka事务）
- **At-Least-Once**：至少一次处理
- **At-Most-Once**：最多一次处理

### 9.6.3 数据安全与隐私

- **数据脱敏**：敏感数据匿名化
- **访问控制**：基于角色的访问控制（RBAC）
- **数据加密**：传输加密、存储加密

## 考点速记

::: tip 高频考点
1. 大数据4V特征：Volume、Velocity、Variety、Value
2. HDFS：NameNode（元数据）、DataNode（数据存储）、Block默认128MB、3副本
3. MapReduce：Map（分而治之）+ Reduce（聚合结果）
4. Spark：内存计算、比MapReduce快10-100倍、RDD抽象
5. NoSQL四类型：键值、文档、列族、图数据库
6. HBase：列族存储、基于HDFS、RowKey字典序
7. Flink：真正的流处理、毫秒级延迟、Exactly-Once语义
8. 数据仓库：面向主题、集成、稳定、历史性；Schema-on-write
9. 数据湖：存储原始数据、Schema-on-read、适合数据探索
10. Lambda架构：批处理层+速度层+服务层；Kappa：纯流处理
:::

## 重点难点

::: warning 重点难点
1. **HDFS读写流程**：写（Client→NameNode→DataNode流水线复制）；读（Client→NameNode获取Block位置→读最近DataNode）
2. **MapReduce Shuffle**：Map输出后分区、排序、合并，Reduce拉取数据后再次排序——这是MapReduce性能瓶颈
3. **Spark RDD容错**：通过Lineage（血缘关系）重建丢失分区，无需数据复制
4. **Exactly-Once语义**：Flink通过Checkpoint（检查点）+ Barrier实现；Kafka通过事务实现
5. **NoSQL选型**：键值（Redis）、文档（MongoDB）、列族（HBase）、图（Neo4j）
6. **数据仓库分层**：ODS（原始）→ DWD（清洗）→ DWS（汇总）→ ADS（应用）
:::
