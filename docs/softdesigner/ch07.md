# 第七章：数据库系统

> 分值占比：10%-15% | 重要程度：★★★★

## 考情分析

数据库系统是软考软件设计师的重要章节，考试重点包括ER模型、关系模型规范化、SQL语言、事务管理和并发控制。需要掌握各种范式的定义与判别方法、事务的ACID特性、封锁协议及并发问题，能够熟练编写SQL查询语句。

## ER模型

### 实体

- **定义**：现实世界中可以区别于其他对象的"事物"或"对象"
- **表示**：矩形框，包含实体名
- **属性**：实体的特征（如学生的学号、姓名、年龄）
- **键**：唯一标识实体的属性或属性集（下划线标注）

### 联系

- **定义**：实体之间的关联关系
- **表示**：菱形框，包含联系名
- **联系类型**：
  - **一对一（1:1）**：一个实体实例至多关联另一个实体的一个实例
  - **一对多（1:n）**：一个实体实例关联多个另一个实体实例
  - **多对多（m:n）**：多个实体实例相互关联多个实体实例
- **联系属性**：联系本身也可以具有属性（如成绩、签约时间）
- **弱实体**：不能独立存在的实体，必须依赖另一个实体（双线矩形）

### ER图表示

- **实体**：矩形
- **属性**：椭圆形，键属性名下划线
- **联系**：菱形
- **关系线**：实体与联系之间的连线，标注基数约束（1, M, N）
- **弱实体集**：双线矩形，识别关系用双线菱形

### ER模型转关系模型

- **实体转关系**：实体名→关系名，属性→关系属性
- **一对一**：可将任一方的主码作为外码加入另一方
- **一对多**：将"多"方的主码作为外码加入"一"方
- **多对多**：创建一个新关系，包含两方的主码和联系的属性

## 关系模型与规范化

### 关系模型基础

- **关系**：一张二维表，表名即关系名
- **元组**：表中的一行，即一个记录
- **属性**：表中的一列，即一个字段
- **域**：属性的取值范围
- **候选码**：能唯一标识元组的最小属性集
- **主码**：从候选码中选择的一个
- **外码**：引用另一关系的主码
- **主属性**：属于任何候选码的属性
- **非主属性**：不属于任何候选码的属性

### 函数依赖

- **定义**：X→Y表示X的值决定Y的值（Y函数依赖于X）
- **完全函数依赖**：X→Y且X的任何真子集都不能决定Y
- **部分函数依赖**：X→Y，但X有真子集X'也能决定Y
- **传递函数依赖**：X→Y，Y→Z，且Y不决定X，则X→Z为传递依赖

### 范式

#### 第一范式（1NF）

- **定义**：所有属性都是不可再分的基本数据项（原子性）
- **要求**：消除重复组，每个单元格只含一个值
- **示例**：一个字段不能同时存"北京,上海,广州"——应拆分为多行或多字段

#### 第二范式（2NF）

- **前提**：满足1NF
- **定义**：消除部分函数依赖（非主属性完全依赖于整个主码）
- **方法**：将部分依赖的非主属性分离到新关系
- **示例**：选课（学号, 课程号, 课程名, 成绩）中，课程名部分依赖于课程号→拆分为选课(学号,课程号,成绩)和课程(课程号,课程名)

#### 第三范式（3NF）

- **前提**：满足2NF
- **定义**：消除传递函数依赖（非主属性不传递依赖于主码）
- **方法**：将传递依赖的非主属性分离
- **示例**：学生(学号,姓名,系名,系地址)中，系地址传递依赖于学号→拆分为学生(学号,姓名,系名)和系(系名,系地址)

#### BCNF（Boyce-Codd范式）

- **前提**：满足3NF
- **定义**：所有非平凡函数依赖X→Y，X必包含候选码
- **与3NF的区别**：3NF允许非主属性传递依赖（如X→Y且Y为主属性），BCNF不允许
- **示例**：选课(学号,课程号,教师)中(课程号,教师)→教室，但(学号,课程号)是候选码→不满足BCNF→需要拆分

### 规范化步骤

1NF → 消除部分依赖 → 2NF
2NF → 消除传递依赖 → 3NF
3NF → 消除所有非主属性对候选码的传递依赖 → BCNF

### 无损联接

- **定义**：关系分解后，通过自然联接能恢复原关系，不产生多余的元组
- **保持函数依赖**：分解后的关系集合的函数依赖集与原关系等价
- **判断方法**：检查分解后的关系是否至少有一个公共属性包含原关系的候选码

### 数据依赖公理（Armstrong公理）

- **自反律**：若Y⊆X，则X→Y
- **传递律**：若X→Y，Y→Z，则X→Z
- **合并律**：若X→Y，X→Z，则X→YZ
- **伪传递律**：若X→Y，WY→Z，则XW→Z
- **分解律**：若X→YZ，则X→Y，X→Z

## SQL语言

### DDL（数据定义语言）

- **CREATE TABLE**：创建表
  ```sql
  CREATE TABLE 表名 (
    列名 数据类型 [约束],
    ...
    [PRIMARY KEY (列名)],
    [FOREIGN KEY (列名) REFERENCES 表名(列名)]
  );
  ```
- **ALTER TABLE**：修改表结构（添加/删除/修改列）
- **DROP TABLE**：删除表

### 数据类型

- **数值型**：INT、BIGINT、SMALLINT、DECIMAL(p,s)、FLOAT、REAL
- **字符型**：CHAR(n)、VARCHAR(n)、TEXT
- **日期时间型**：DATE、TIME、DATETIME、TIMESTAMP
- **布尔型**：BOOLEAN

### DML（数据操作语言）

- **INSERT**：插入数据
  ```sql
  INSERT INTO 表名 (列1,列2) VALUES (值1,值2);
  ```
- **UPDATE**：更新数据
  ```sql
  UPDATE 表名 SET 列=值 WHERE 条件;
  ```
- **DELETE**：删除数据
  ```sql
  DELETE FROM 表名 WHERE 条件;
  ```
- **SELECT**：查询数据（见连接查询、子查询）

### DCL（数据控制语言）

- **GRANT**：授予权限
  ```sql
  GRANT SELECT,INSERT ON 表名 TO 用户;
  ```
- **REVOKE**：回收权限
  ```sql
  REVOKE SELECT ON 表名 FROM 用户;
  ```

### 连接查询

#### 内连接（INNER JOIN）

- 返回两个表中满足连接条件的元组
- 等价于对两表笛卡尔积后筛选连接条件

#### 外连接

- **左外连接（LEFT JOIN）**：返回左表所有元组，右表匹配不到则为NULL
- **右外连接（RIGHT JOIN）**：返回右表所有元组，左表匹配不到则为NULL
- **全外连接（FULL JOIN）**：返回两表所有元组，不匹配处为NULL

#### 交叉连接（CROSS JOIN）

- 返回两表的笛卡尔积（每个元组组合）
- 等价于不加WHERE条件的多表FROM

#### 自连接

- 同一张表当作两张表使用，通过别名区分
- 常用于层次查询（如员工与经理关系）

### 子查询

- **WHERE子查询**：条件中嵌套SELECT
  - 单行子查询：用 =, >, <, >=, <=, != 比较
  - 多行子查询：用 IN, NOT IN, ANY, ALL
- **FROM子查询（派生表）**：子查询结果作为临时表
- **SELECT子查询（标量子查询）**：返回单一值
- **EXISTS子查询**：判断子查询是否有结果，有则TRUE

## 事务管理

### ACID特性

- **原子性（Atomicity）**：事务是不可分割的最小单位，要么全部成功，要么全部失败
- **一致性（Consistency）**：事务执行前后数据库处于一致性状态
- **隔离性（Isolation）**：并发事务之间相互隔离，互不干扰
- **持久性（Durability）**：事务一旦提交，其修改永久有效

### 并发问题

- **丢失更新（Lost Update）**：两个事务同时读取同一数据，先后写入，后写入覆盖前写入
- **脏读（Dirty Read）**：事务读取了另一个未提交事务修改的数据
- **不可重复读（Non-repeatable Read）**：同一事务中两次读取同一数据得到不同结果（被其他事务修改并提交）
- **幻读（Phantom Read）**：同一事务中两次查询，第二次结果集比第一次多（其他事务插入了新行）

### 封锁协议

#### 一级封锁协议

- 事务T在修改数据前必须先加X锁（排他锁），事务结束后释放
- **防止**：丢失更新
- **不能防止**：脏读、不可重复读、幻读

#### 二级封锁协议

- 在一级基础上增加：事务T在读取数据前必须加S锁（共享锁），读完立即释放
- **防止**：丢失更新、脏读
- **不能防止**：不可重复读、幻读

#### 三级封锁协议

- 在二级基础上增加：事务T在读取数据前加S锁，事务结束后才释放
- **防止**：丢失更新、脏读、不可重复读
- **不能防止**：幻读

#### 两段锁协议（2PL）

- **扩展阶段**：事务可以申请新锁，不能释放任何锁
- **收缩阶段**：事务可以释放锁，不能申请新锁
- **保证**：可串行化（冲突可串行化）
- **与三级封锁关系**：三级封锁协议是两段锁协议的特例

### 封锁类型

| 锁类型 | 加锁条件 | 兼容性 | 目的 |
|-------|---------|-------|------|
| 共享锁（S锁） | 读操作 | 与S兼容，与X不兼容 | 防止脏读 |
| 排他锁（X锁） | 写操作 | 与S、X都不兼容 | 防止丢失更新 |

### 隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|----------|------|
| 读未提交（Read Uncommitted） | 可能 | 可能 | 可能 |
| 读已提交（Read Committed） | 防止 | 可能 | 可能 |
| 可重复读（Repeatable Read） | 防止 | 防止 | 可能 |
| 可串行化（Serializable） | 防止 | 防止 | 防止 |

## 数据库安全与恢复

### 数据库安全

- **用户权限管理**：GRANT/REVOKE控制访问权限
- **视图**：虚表，提供数据安全性和逻辑独立性
  - 隐藏敏感数据（只暴露非敏感列）
  - 简化复杂查询
- **完整性约束**：
  - **实体完整性**：主码不能为NULL，必须唯一
  - **参照完整性**：外码要么是引用表的主码，要么为NULL
  - **用户定义完整性**：业务规则约束（如年龄>0）
- **触发器**：事件驱动的自动执行程序
  - INSERT/UPDATE/DELETE时触发
  - 用于强制业务规则和数据审计

### 数据库恢复

- **故障类型**：
  - **事务故障**：逻辑错误（死锁、约束冲突）
  - **系统故障**：系统崩溃（掉电、软件错误）
  - **介质故障**：存储介质损坏（磁盘坏道）
- **恢复技术**：
  - **日志文件（Log）**：记录所有更新操作，用于故障恢复
  - **数据库镜像**：实时备份，用于介质故障恢复
  - **检查点（Checkpoint）**：定期保存状态，减少恢复时间
- **恢复策略**：
  - **事务故障**：反向扫描日志，撤销（UNDO）未完成事务
  - **系统故障**：正向扫描重做（REDO）已提交事务，反向扫描撤销未完成事务
  - **介质故障**：从最近备份恢复 + 重做已提交事务

### 数据库设计步骤

1. **需求分析**：收集用户需求，生成数据字典
2. **概念结构设计**：设计ER模型
3. **逻辑结构设计**：ER转关系模型，规范化设计
4. **物理结构设计**：设计存储结构、索引
5. **数据库实施**：建立数据库、加载数据、编写应用
6. **数据库运行与维护**：监控、调优、备份

## 考点速记

::: tip 高频考点
1. ER模型：实体矩形、联系菱形、属性椭圆；一对一/一对多/多对多转换规则
2. 范式判别：1NF（原子性）→2NF（消除部分依赖）→3NF（消除传递依赖）→BCNF（所有非平凡FD的左边含候选码）
3. 规范化方法：部分依赖→拆出新关系；传递依赖→拆出新关系
4. 事务ACID：原子性、一致性、隔离性、持久性——缺一不可
5. 并发问题：脏读（读未提交）、不可重复读（两次读到不同）、幻读（多出幻影行）
6. 隔离级别：RC防止脏读，RR防止不可重复读，Serializable全部防止
7. 封锁协议：一级防丢失更新，二级防脏读，三级防不可重复读，2PL保证可串行化
8. SQL：JOIN（INNER/LEFT/RIGHT/FULL），子查询（IN/EXISTS/ANY/ALL）
9. 无损联接：分解后至少有一个关系包含原候选码
10. 数据库恢复：UNDO撤销未完成，REDO重做已提交；检查点减少恢复时间
:::

## 重点难点

::: warning 重点难点
1. **范式判别步骤**：
   - 第一步：判断1NF（是否原子）
   - 第二步：找主码，检查非主属性是否部分依赖→部分依赖则非2NF
   - 第三步：检查非主属性是否传递依赖→传递依赖则非3NF
   - 第四步：检查所有非平凡FD的左边是否含候选码→不含则非BCNF
2. **ER模型转关系模型规则**：
   - 实体→关系（键→主码）
   - 1:1：任一方加对方主码为外码
   - 1:n：n方加1方主码为外码
   - m:n：新建关系，包含两方主码+联系属性
3. **封锁协议与并发问题对应**：
   - 一级（X锁+释放）：丢失更新✅，脏读❌
   - 二级（X锁+S锁+立即释放S锁）：丢失更新✅，脏读✅，不可重复读❌
   - 三级（X锁+S锁+事务结束释放）：丢失更新✅，脏读✅，不可重复读✅，幻读❌
   - 可串行化（Serializable）：全部✅
4. **内连接 vs 外连接**：
   - INNER JOIN：只保留两表都匹配的行
   - LEFT JOIN：保留左表所有行，右表不匹配则为NULL
   - 注意：CROSS JOIN是笛卡尔积，没有WHERE条件的多表查询
5. **事务恢复的UNDO/REDO**：
   - UNDO：回滚事务修改（删除日志中的更新记录）
   - REDO：重新执行已提交事务（重复日志中的更新）
   - 先UNDO未完成事务，再REDO已提交事务（WAL先写日志后写数据）
:::
