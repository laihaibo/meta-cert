# 第六章：面向对象分析与设计

> 分值占比：10%-15% | 重要程度：★★★★★

## 考情分析

面向对象分析与设计是软考中级软件设计师的核心章节，考试重点包括UML建模图的识别与绘制、面向对象设计原则、设计模式的应用场景、软件架构风格。需要掌握各种UML图的符号含义和适用场景，理解常见设计模式的结构与用途，能够根据需求选择合适的架构模式。

## UML建模

### 用例图（Use Case Diagram）

- **用途**：描述系统与外部参与者（Actor）的交互，表达系统的功能需求
- **元素**：
  - **参与者**：与系统交互的外部实体（人、系统、设备）
  - **用例**：系统提供的功能单元（椭圆表示）
  - **关系**：
    - 关联：参与者与用例之间
    - 包含（include）：一个用例必须包含另一个用例
    - 扩展（extend）：一个用例可选地扩展另一个用例
    - 泛化：用例之间的继承关系
- **应用场景**：需求分析阶段，与用户沟通功能需求

### 类图（Class Diagram）

- **用途**：描述系统中类的结构及类之间的关系
- **元素**：
  - **类**：三层结构（类名、属性、方法）
  - **关系**：
    - **关联**：A类知道B类，单向或双向（普通箭头线）
    - **聚合**：整体-部分关系，部分可独立存在（空心菱形）
    - **组合**：整体-部分关系，部分不能独立存在（实心菱形）
    - **依赖**：一个类使用另一个类（虚线箭头）
    - **泛化/继承**：子类-父类关系（空心三角箭头）
    - **实现**：接口-实现类关系（虚线空心三角箭头）
- **可见性修饰符**：+（public）、-（private）、#（protected）、~（package）

### 序列图（Sequence Diagram）

- **用途**：描述对象之间消息传递的时间顺序
- **元素**：
  - **对象/角色**：顶部矩形框
  - **生命线**：对象下方的虚线
  - **激活条**：对象执行操作的时期
  - **消息**：对象之间的箭头（同步、异步、返回）
- **特点**：强调消息的时间顺序，适合描述交互场景

### 状态图（State Diagram）

- **用途**：描述一个对象在其生命周期内的状态变化
- **元素**：
  - **状态**：圆角矩形
  - **初态**：实心圆点
  - **终态**：带圈的实心圆点
  - **转移**：状态之间的箭头，标注触发事件和动作
- **适用**：对象有明确的状态变化（如订单状态流转）

### 活动图（Activity Diagram）

- **用途**：描述业务流程或操作的工作流程
- **元素**：
  - **活动**：圆角矩形
  - **开始/结束**：实心圆点/带圈圆点
  - **决策节点**：菱形，条件分支
  - **合并节点**：菱形，分支汇合
  - **泳道**：划分不同执行者或对象负责的活动
- **与流程图区别**：支持并行活动（分叉/汇合）

### 组件图（Component Diagram）

- **用途**：描述系统的物理结构，组件之间的依赖关系
- **元素**：
  - **组件**：带标签的矩形（带接口端口）
  - **接口**：带名称的圆圈或半圆
  - **依赖关系**：组件之间的连线
- **适用**：系统架构设计，模块划分

### 部署图（Deployment Diagram）

- **用途**：描述系统的物理部署结构，节点及节点上的组件
- **元素**：
  - **节点**：设备或执行环境（服务器、设备图标）
  - **组件**：部署在节点上的软件模块
  - **连接**：节点之间的通信路径
- **适用**：分布式系统部署设计

## 面向对象分析

### 需求分析

- **目标**：理解用户需求，确定系统功能和非功能需求
- **方法**：
  - 访谈、问卷、现场观察
  - 用例驱动：识别参与者和用例
- **产出**：需求规格说明书（SRS）

### OOA方法

- **Booch方法**：
  - 强调对象模型、动态模型、功能模型
  - 使用类图、状态图、模块图
- **OMT方法（对象建模技术）**：
  - 三种模型：对象模型、动态模型、功能模型
  - 对象模型 → 类图，动态模型 → 状态图/活动图
- **OOSE方法（面向对象软件工程）**：
  - 引入用例概念，强调用例驱动
- **统一过程（UP/RUP）**：
  - 四个阶段：初始、细化、构造、移交
  - 核心工作流：需求、分析、设计、实现、测试
  - 迭代增量开发

## 面向对象设计

### 设计原则（SOLID）

- **S — 单一职责原则（SRP）**：一个类只负责一个职责
- **O — 开闭原则（OCP）**：对扩展开放，对修改关闭
- **L — 里氏替换原则（LSP）**：子类必须能替换父类而不改变程序正确性
- **I — 接口隔离原则（ISP）**：客户端不应依赖它不需要的接口
- **D — 依赖倒置原则（DIP）**：高层模块不依赖低层模块，两者都依赖抽象

### 设计模式分类（GoF 23种）

#### 创建型模式（5种）

| 模式 | 意图 | 适用场景 |
|-----|------|---------|
| 单例模式（Singleton） | 确保一个类只有一个实例 | 全局配置、连接池、日志 |
| 工厂方法模式（Factory Method） | 定义创建对象的接口，延迟到子类 | 产品种类不确定，需灵活扩展 |
| 抽象工厂模式（Abstract Factory） | 创建相关产品族的接口 | 跨平台UI、多数据库支持 |
| 建造者模式（Builder） | 分步骤构建复杂对象 | 构建复杂配置对象（如Builder） |
| 原型模式（Prototype） | 通过复制现有对象创建新对象 | 创建成本高的对象（如深拷贝） |

#### 结构型模式（7种）

| 模式 | 意图 | 适用场景 |
|-----|------|---------|
| 适配器模式（Adapter） | 接口转换，使不兼容的接口协同工作 | 旧系统集成、接口转换 |
| 桥接模式（Bridge） | 将抽象与实现解耦 | 多维度变化（如跨平台+多格式） |
| 组合模式（Composite） | 将对象组合成树形结构 | 文件系统、GUI组件树 |
| 装饰模式（Decorator） | 动态为对象添加职责 | IO流、动态功能扩展 |
| 外观模式（Facade） | 提供统一接口访问子系统 | 简化复杂系统调用 |
| 享元模式（Flyweight） | 共享细粒度对象，减少内存 | 大量相似对象（如文字处理器字符） |
| 代理模式（Proxy） | 为其他对象提供代理以控制访问 | 远程代理、虚拟代理、保护代理 |

#### 行为型模式（11种）

| 模式 | 意图 | 适用场景 |
|-----|------|---------|
| 策略模式（Strategy） | 定义算法族，使它们可互换 | 多种排序/支付策略可切换 |
| 观察者模式（Observer） | 定义一对多依赖，状态变化时通知所有依赖者 | GUI事件处理、消息通知 |
| 模板方法模式（Template Method） | 定义算法骨架，子类实现具体步骤 | 框架设计、流程标准化 |
| 责任链模式（Chain of Responsibility） | 请求沿链传递，直到被处理 | 审批流程、过滤器链 |
| 状态模式（State） | 对象状态变化时改变行为 | 订单状态、TCP连接状态 |
| 迭代器模式（Iterator） | 顺序访问聚合对象元素，不暴露内部结构 | 集合遍历、JDK集合框架 |
| 访问者模式（Visitor） | 作用于对象结构的操作，不改变结构 | 报表系统、编译器AST |
| 备忘录模式（Memento） | 保存对象状态，可恢复 | 撤销/恢复、游戏存档 |
| 命令模式（Command） | 将请求封装为对象 | 事务、宏命令、队列请求 |
| 解释器模式（Interpreter） | 定义语言文法，解释执行 | 规则引擎、正则表达式 |
| 中介者模式（Mediator） | 用中介对象封装多对象交互 | 聊天室、GUI事件分发 |

## 软件架构设计

### 分层架构

- **三层架构**：表示层（UI）、业务逻辑层（BLL）、数据访问层（DAL）
- **n层架构**：在三层基础上进一步拆分（如添加服务层、缓存层）
- **优点**：层次清晰、易维护、可复用
- **缺点**：性能开销（跨层调用）

### MVC架构

- **M（Model）**：数据模型，管理数据和业务逻辑
- **V（View）**：视图，负责UI展示
- **C（Controller）**：控制器，处理用户输入，协调Model和View
- **变体**：
  - **MVP**：Presenter代替Controller，View被动依赖Presenter
  - **MVVM**：ViewModel双向绑定View，数据变化自动更新UI

### 微服务架构

- **定义**：将单一应用拆分为多个小型独立服务，通过API通信
- **特点**：
  - 每个服务独立开发、部署、扩展
  - 服务间通过REST API或消息队列通信
  - 每个服务有自己的数据库
- **优点**：技术异构、独立扩展、故障隔离、团队自治
- **挑战**：分布式系统复杂性、数据一致性、服务发现、链路追踪

### 架构风格

| 架构风格 | 特点 | 适用场景 |
|---------|------|---------|
| 管道-过滤器 | 数据流经一系列处理组件 | 编译器、信号处理 |
| 事件驱动 | 组件通过事件通信 |  GUI、异步系统 |
| 仓库风格 | 共享数据仓库+独立处理组件 | 数据库系统、黑板系统 |
| C/S | 客户端-服务器 | 桌面应用、企业系统 |
| B/S | 浏览器-服务器 | Web应用 |
| P2P | 点对点，无中心服务器 | 文件共享、区块链 |

## 考点速记

::: tip 高频考点
1. UML图识别：用例图（参与者和功能）、类图（类及其关系）、序列图（时序）
2. 关联/聚合/组合三者的区别：组合是强聚合（生命周期绑定）
3. SOLID原则口诀：单一开闭里氏接依（SRP→OCP→LSP→ISP→DIP）
4. 创建型5种、结构型7种、行为型11种，共23种GoF设计模式
5. 工厂方法vs抽象工厂：前者生产单一产品，后者生产产品族
6. MVC三组件：Model数据、View展示、Controller控制
7. 微服务特点：独立部署、独立数据库、API通信、技术异构
8. 单例模式两种实现：饿汉式（类加载时创建）、懒汉式（首次使用时创建）
:::

## 重点难点

::: warning 重点难点
1. **UML关系辨析**：
   - 关联：连接两个类，有方向
   - 聚合（空心菱形）：整体-部分，部分可独立
   - 组合（实心菱形）：整体-部分，部分不可独立，生命周期绑定
   - 依赖（虚线箭头）：一个类使用另一个类，最弱关系
2. **策略模式 vs 状态模式**：
   - 策略：算法可替换，由客户端选择策略
   - 状态：对象自身状态变化自动切换行为
   - 两者结构相似但意图完全不同
3. **代理模式类型**：
   - 远程代理：隐藏网络通信细节
   - 虚拟代理：延迟加载（如图片占位符）
   - 保护代理：控制访问权限
4. **三层架构 vs MVC**：
   - 三层：按功能层次划分（UI/BLL/DAL）
   - MVC：按职责划分（M/V/C），V和M不直接交互
5. **微服务 vs SOA**：
   - SOA：粗粒度服务，ESB总线，重量级
   - 微服务：细粒度服务，轻量级通信，去中心化
:::
