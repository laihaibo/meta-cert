# 第五章：软件工程

> 分值占比：10%-12% | 重要程度：★★★★

## 考情分析

本章是软件设计师考试的重点章节，主要介绍软件开发过程、软件设计方法、软件测试、项目管理等。考试重点在于软件开发模型、软件设计原则、测试用例设计和项目管理知识。

## 5.1 软件工程概述

### 5.1.1 软件工程定义

软件工程是将系统化的、规范化的、可量化的方法应用于软件的开发、运行和维护的过程，即将工程化应用于软件开发。

**软件工程三要素：**
- **方法**：完成软件开发任务的技术方法
- **工具**：支持软件开发过程的软件工具
- **过程**：支持软件开发各个阶段的管理和控制

### 5.1.2 软件生命周期

**定义：** 软件产品从提出、开发、使用到维护直到废弃的整个过程。

**软件生命周期各阶段：**

| 阶段 | 主要任务 | 输出文档 |
|------|---------|---------|
| 可行性研究 | 技术、经济、社会可行性分析 | 可行性研究报告 |
| 需求分析 | 确定系统功能和性能需求 | 软件需求规格说明书（SRS） |
| 概要设计 | 系统架构设计、模块划分 | 概要设计说明书 |
| 详细设计 | 模块内部算法和数据结构 | 详细设计说明书 |
| 编码 | 编写程序代码 | 源程序 |
| 测试 | 单元测试、集成测试、系统测试 | 测试报告 |
| 运行维护 | 纠错、适应、完善 | 维护记录 |

### 5.1.3 软件开发模型

**瀑布模型：**
- 线性顺序，各阶段无交叉
- 优点：简单、易于管理
- 缺点：不灵活，后期变更代价大
- 适用：需求明确、稳定的项目

**原型模型：**
- 快速构建原型，用户反馈后迭代
- 优点：用户参与，需求理解准确
- 缺点：原型可能被误认为最终产品
- 适用：需求不确定、需要探索的项目

**增量模型：**
- 分批次交付功能增量
- 优点：可并行开发，早期获得反馈
- 缺点：需要良好的软件架构
- 适用：大型复杂系统

**螺旋模型：**
- 结合瀑布和原型，强调风险分析
- 每个周期包含目标设定、风险评估、开发验证、客户评估
- 优点：风险驱动，适合大型高风险项目
- 缺点：管理复杂，需要风险评估经验

**敏捷开发模型：**
- 迭代增量，拥抱变化
- 代表：Scrum、XP、极限编程
- 优点：快速响应变化，用户满意度高
- 缺点：文档少，需团队经验丰富

## 5.2 软件需求分析

### 5.2.1 需求分析任务

- **确定系统功能需求**：系统应该做什么
- **确定系统非功能需求**：性能、可靠性、安全性等
- **建立分析模型**：数据流图、实体关系图等
- **编写需求规格说明书**：SRS文档

### 5.2.2 结构化分析方法（SA）

**核心思想：** 自顶向下、逐层分解

**主要工具：**
- **数据流图（DFD）**：描述数据在系统中的流动和处理
  - 外部实体：系统外的人或物
  - 加工（处理）：数据变换
  - 数据存储：保存数据
  - 数据流：数据流动方向
- **数据字典（DD）**：定义DFD中所有元素
- **加工逻辑说明**：描述加工的处理逻辑

### 5.2.3 需求管理

- **需求获取**：访谈、问卷、观察、原型
- **需求分析**：分类、建模、优先级排序
- **需求规格说明**：编写SRS文档
- **需求验证**：评审、测试用例
- **需求变更控制**：变更申请、评估、批准、实施

## 5.3 软件设计

### 5.3.1 概要设计

**概要设计主要内容：**
- **设计软件系统结构**：划分模块，确定模块间关系
- **数据结构设计**：数据库设计、文件设计
- **接口设计**：人机界面、软件接口、硬件接口
- **运行设计**：运行模块组合、运行控制
- **出错处理设计**：异常处理、容错设计

**模块设计原则（高内聚低耦合）：**
- **内聚性（Cohesion）：** 模块内部各元素结合的紧密程度
  - 高→低：功能内聚、顺序内聚、通信内聚、过程内聚、时间内聚、逻辑内聚、偶然内聚
- **耦合性（Coupling）：** 模块间连接的紧密程度
  - 低→高：非直接耦合、数据耦合、标记耦合、控制耦合、外部耦合、公共耦合、内容耦合

### 5.3.2 详细设计

**详细设计工具：**
- **程序流程图（PFC）**：图形表示程序逻辑
- **N-S图（盒图）**：结构化流程图，无任意跳转
- **PAD图（问题分析图）**：树形结构，层次清晰
- **伪代码（PDL）**：类语言描述，介于语言和自然语言之间

### 5.3.3 面向对象设计

**UML（统一建模语言）：**

**常用图：**
| 图类型 | 作用 | 示例 |
|--------|------|------|
| 用例图 | 描述系统功能和参与者 | 用户登录用例 |
| 类图 | 描述类及其关系 | 继承、关联、聚合 |
| 对象图 | 描述对象及其关系 | 某时刻对象实例 |
| 时序图 | 描述对象间消息传递顺序 | 登录流程时序 |
| 协作图 | 描述对象间协作关系 | 对象交互网络 |
| 状态图 | 描述对象状态变化 | 订单状态机 |
| 活动图 | 描述工作流程 | 业务流程 |
| 构件图 | 描述软件组件 | 模块依赖 |
| 部署图 | 描述物理部署 | 网络拓扑 |

**设计模式（GoF 23种）：**

**创建型模式：**
- 单例（Singleton）：保证唯一实例
- 工厂方法（Factory Method）：定义创建接口
- 抽象工厂（Abstract Factory）：创建相关对象族
- 建造者（Builder）：分步构造复杂对象
- 原型（Prototype）：通过复制创建对象

**结构型模式：**
- 适配器（Adapter）：转换接口
- 桥接（Bridge）：分离抽象和实现
- 组合（Composite）：树形结构
- 装饰（Decorator）：动态添加功能
- 外观（Facade）：提供统一接口
- 享元（Flyweight）：共享细粒度对象
- 代理（Proxy）：控制对象访问

**行为型模式：**
- 观察者（Observer）：事件通知
- 策略（Strategy）：算法族封装
- 命令（Command）：请求封装为对象
- 迭代器（Iterator）：顺序访问集合
- 模板方法（Template Method）：定义算法骨架
- 责任链（Chain of Responsibility）：请求传递链

## 5.4 软件测试

### 5.4.1 测试基础

**测试原则：**
- 尽早测试、持续测试
- 测试应基于需求
- 不可能进行穷尽测试
- 测试应兼顾功能和结构
- 测试应防错而非证对
- 测试用例应包含有效和无效输入

**测试信息流：**
- 测试用例：输入 + 预期输出
- 测试结果：实际输出
- 比较：实际输出 vs 预期输出

### 5.4.2 测试用例设计

**白盒测试（结构测试）：**
- 基于程序内部结构设计测试用例
- **覆盖标准（从弱到强）：**
  1. 语句覆盖：每条语句至少执行一次
  2. 判定覆盖（分支覆盖）：每个判定的每个分支至少执行一次
  3. 条件覆盖：每个条件的每个可能值至少出现一次
  4. 判定-条件覆盖：同时满足2和3
  5. 条件组合覆盖：每个条件的所有组合至少出现一次
  6. 路径覆盖：每条路径至少执行一次

**黑盒测试（功能测试）：**
- 基于功能规格设计测试用例，不考虑内部结构
- **常用方法：**
  - 等价类划分：有效等价类、无效等价类
  - 边界值分析：取边界值、略大于/小于边界
  - 因果图：分析输入条件组合
  - 错误推测法：基于经验推测错误

### 5.4.3 测试阶段

| 测试阶段 | 测试对象 | 测试内容 | 测试方法 |
|---------|---------|---------|---------|
| 单元测试 | 模块 | 模块功能、接口 | 白盒为主 |
| 集成测试 | 模块组合 | 接口、数据传递 | 黑盒为主 |
| 确认测试 | 完整系统 | 功能、性能 | 黑盒 |
| 系统测试 | 完整系统 | 功能、性能、安全等 | 黑盒 |
| 验收测试 | 完整系统 | 用户需求符合度 | 黑盒 |

**集成测试方法：**
- **非增量式（一次性）**：先分别测试，再集成
- **增量式：**
  - 自顶向下：先测试顶层模块，逐步向下
  - 自底向上：先测试底层模块，逐步向上
  - 三明治：结合自顶向下和自底向上

### 5.4.4 调试（排错）

**调试策略：**
- 蛮力法：穷举所有可能
- 回溯法：从错误处回溯找原因
- 原因排除法：二分法、归纳法、演绎法
- 测试法：设计测试用例定位错误

## 5.6 软件项目管理

### 5.6.1 进度管理

**Gantt图（甘特图）：**
- 横轴表示时间，纵轴表示任务
- 直观表示任务进度和依赖关系

**PERT图（计划评审技术）：**
- 有向图表示任务流程
- 节点表示里程碑，边表示任务
- 用于计算关键路径

**关键路径法（CPM）：**
- 关键路径：决定项目最早完成时间的路径
- 关键活动：关键路径上的活动
- 总时差 = 最迟完成时间 - 最早完成时间
- 关键活动总时差为0

### 5.6.2 软件质量

**质量特性（McCall）：**
- 产品运行：正确性、可靠性、效率、完整性、可用性
- 产品修改：可维护性、灵活性、可测试性
- 产品转移：可移植性、可复用性、互操作性

**CMM/CMMI（能力成熟度模型）：**
- **初始级**：过程无序，成功依赖个人
- **可重复级**：建立基本项目管理过程
- **已定义级**：建立标准过程和定义
- **已管理级**：建立量化质量目标
- **优化级**：持续过程改进

### 5.6.3 软件度量

**规模度量：**
- **代码行（LOC/LSLOC）**：源代码行数
- **功能点（FP）**：基于功能点的度量

**McCabe复杂度：**
- 基于控制流的圈复杂度
- V(G) = E - N + 2P（边数 - 节点数 + 2×连通分量数）
- V(G) = 判定节点数 + 1
- V(G)越大，程序越复杂，测试越困难
